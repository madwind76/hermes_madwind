# 리눅스 프로세스 메모리 속 SSH 패스워드 복구 (Linux SSHD Process Memory dump)

> **난이도**: 초중급  
> **소요 시간**: 25~30분  
> **참고 picoCTF 문제**: 프로세스 가상 메모리 크리덴셜 복구 (Linux SSHD Memory 분석)

## 1. 배경 시나리오
사내 자산 유출 사고 조사를 진행하던 중, 악성 행위자가 원격에서 리눅스 터미널 세션에 로그온해 활동 중인 정황이 확인되었습니다. 침해 대응팀은 침입 차단 전, 용의자가 로그인 시 사용했던 사용자 계정 암호를 탈취하여 추후 자격증명 재사용(Credential Reuse) 분석을 수행하고자 실시간 원격 데스크톱 세션을 제어하는 **SSHD (SSH Daemon)** 프로세스의 가상 메모리를 강제로 덤프했습니다. 덤프하여 획득한 `sshd_memory.dmp` 파일을 분석하여, **인증 처리 도중 SSHD 메모리 힙(Heap) 또는 스택 영역에 평문으로 임시 적재되어 잔재하던 패스워드(플래그)**를 복구하십시오.

## 2. 제공 파일
* `sshd_memory.dmp` (인증 세션이 수립된 시점의 `sshd` 특정 서브 프로세스 가상 메모리 공간에서 덤프한 바이너리 파일)

## 3. 문제 목표
리눅스의 SSHD 데몬이 원격 사용자 자격증명 인증을 위해 PAM(Pluggable Authentication Modules) 라이브러리와 통신할 때, 메모리 상에 사용자 계정명과 비밀번호를 평문 구조체 형태로 임시 보존하는 구간이 존재함을 파악하고, 프로세스 덤프 바이너리에서 해당 크리덴셜 영역을 격리/추출합니다.

## 4. 의도한 풀이 흐름
1. **프로세스 덤프 바이너리 진단**:
   * 제공된 `sshd_memory.dmp`는 프로세스의 특정 메모리 맵 세그먼트 자체를 덤프한 것입니다.
   * 메모리 상에서 아스키 가독 문자열 목록을 선제 추출하기 위해 `strings` 명령어를 가동하고 플래그 포맷을 검색합니다:
     ```bash
     strings sshd_memory.dmp | grep "picoCTF"
     ```
2. **패스워드 구조체 흔적 발견**:
   * 검색 결과, SSHD 인증 서브 프로세스의 메모리 힙 버퍼 세그먼트 부근에서 아래와 같이 사용자 명칭과 패스워드가 연이어 나열된 메모리 적재 스트림을 검출합니다:
     `sshdauth: user=sec-admin pass=picoCTF{sshd_pl41nt3xt_pwd_r3cov3ry_m3m}`
3. **플래그 도출**:
   * 계정명 우측의 암호 필드 데이터로 임시 수록되어 있던 최종 플래그 값을 획득합니다:
     `picoCTF{sshd_pl41nt3xt_pwd_r3cov3ry_m3m}`

## 5. 정답(플래그) 규칙 및 예시
* **플래그 형식**: `picoCTF{<flag_string>}`
* **예시**: `picoCTF{sshd_pl41nt3xt_pwd_r3cov3ry_m3m}`

## 6. 출제자 노트 & 파일 제작 가이드
* **아티팩트 제작 방법**:
  1. 테스트 리눅스 서버에서 SSH 데몬을 가동합니다.
  2. 클라이언트 PC에서 테스트 계정(예: sec-admin)과 플래그 비밀번호 `picoCTF{sshd_pl41nt3xt_pwd_r3cov3ry_m3m}`를 입력하여 RDP/SSH 접속을 시작합니다.
  3. 로그인 성공 시점 또는 인증 완료 직후, 원격 접속을 대기 처리 중인 하위 `sshd: sec-admin@pts/X` 인증 핸들러 프로세스의 PID를 조회합니다:
     `ps aux | grep "sshd:"`
  4. 디버거(GDB) 또는 메모리 덤프 스크립트를 사용하여 해당 프로세스의 가상 메모리(/proc/<PID>/mem) 내 힙(Heap) 세그먼트 주소 영역을 바이너리 파일로 강제 저장합니다:
     `gcore -o sshd_memory <PID>` (gcore 도구를 이용하면 해당 프로세스의 전체 가상 메모리를 통째로 코어 덤프할 수 있습니다)
  5. 덤프 결과물인 코어 덤프 파일 내에서 `picoCTF` 키워드가 기입된 페이지 섹터를 별도로 격리 분할하여 `sshd_memory.dmp` 파일로 정제해 배포합니다.
* **출제 포인트**: 
  * 리눅스 메모리 포렌식(Linux Memory Forensics)의 핵심인 활성 프로세스 메모리 덤프 파싱 및 덤프 기법(procfs 활용)의 실무 유용성을 학습시키고, SSHD 등 고권한 통신 세션 분석 시 런타임 메모리 가독을 통한 계정 정보 역추적의 파급력을 인식시킵니다.

## 7. 트러블슈팅 및 힌트
* **Q. /proc/<PID>/mem 파일은 일반 cat 명령으로 왜 읽을 수 없나요?**
  * A. `/proc/<PID>/mem` 파일은 프로세스의 가상 주소 공간 전체를 1대1 매핑하여 추상화한 특수 가상 파일입니다. 프로세스가 할당받지 않은 잘못된 주소(비어 있는 주소) 영역에 접근하여 읽기를 시도하면 커널이 즉시 I/O 에러를 리턴하며 차단합니다. 따라서 해당 프로세스의 가상 메모리 맵 목록 정보인 `/proc/<PID>/maps` 파일을 먼저 파싱하여, 실제로 메모리가 맵핑(Readable)되어 있는 힙, 스택, 라이브러리 주소 대역(오프셋)만 지정해 `dd` 명령어로 부분 바이트 카빙을 가동해야 안전하게 덤프할 수 있습니다.
* **Q. SSHD 프로세스가 패스워드를 계속 평문으로 메모리에 남겨 두는 것은 취약점인가요?**
  * A. SSHD는 클라이언트가 전달한 평문 비밀번호 값을 시스템 로컬 PAM 스택에 전달해 대조 결과를 받아오기 전까지 한시적으로 힙 메모리에 적재해 처리해야 합니다. 처리 직후 윈도우/리눅스 보안 패치 규정에 의해 메모리를 제로화(Zeroization)하도록 권고되지만, 비정상적 가동이나 메모리 파편화(Fragmentation)로 인해 해제된 힙 슬랙 공간(Heap Slack)에 자격증명 평문 흔적이 고스란히 남아 있는 케이스가 침해 대응 현장에서 자주 발견됩니다.

## 8. 학습 포인트
* **리눅스 가상 메모리 체계(/procfs)**: 프로세스의 가상 메모리 맵핑 정보(`maps`) 및 메모리 접근 파일(`mem`)의 상관관계와 메모리 복원 이론을 완벽히 이해합니다.
* **프로세스 덤프 분석**: 실행 프로세스의 비파괴 감사(Memory Core Dump / Gcore) 절차를 훈련하고, 기밀 평문 데이터의 힙 영역 잔재성을 검증하는 침해 추적 기법을 마스터합니다.
