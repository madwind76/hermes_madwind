# 윈도우 바탕화면 바로가기 파일의 타깃 컴파일 일시 추출 (LNK File Target Creation Timestamp)

> **난이도**: 초급  
> **소요 시간**: 15~20분  
> **참고 picoCTF 문제**: 삭제된 실행 파일 원본 타임라인 규명 (Windows LNK Target Timestamps 분석)

## 1. 배경 시나리오
공격자가 원격 화면 제어를 시도하고 도구 흔적을 인멸하기 위해 자신의 공격 실행 파일(`attack_tool.exe`)을 디스크 상에서 안전 소거(Wipe)하여 삭제했습니다. 마스터 파일 테이블(MFT) 상의 인덱스가 덮어씌워져 이진 파일의 속성 복구가 불가능한 상태입니다. 하지만 피의자의 최근 사용 기록 목록을 검토하던 중, 해당 파일 실행 당시에 생성되었던 단축 바로가기 파일인 `evidence.lnk` 아티팩트를 확보하는 데 성공했습니다. 윈도우 바로가기(`.lnk`) 링크 파일 헤더에는 바로가기 파일 자체의 생성 시각뿐만 아니라, **링크가 걸려있던 대상 타깃 실행 파일(`attack_tool.exe`)의 원본 생성 시각(Target Creation Time)** 메타데이터가 영구 보존됩니다. 확보한 LNK 파일을 분석하여, **삭제 유실되었던 공격 도구의 실제 타깃 생성 시각(플래그)**을 특정하십시오.

## 2. 제공 파일
* `evidence.lnk` (수집한 윈도우 셸 링크 바로가기 파일)

## 3. 문제 목표
윈도우 바로가기 링크 파일 헤더 구조(특히 **ShellLinkHeader** 내에 정의된 **Target CreationTime** 타임스탬프 필드 정보의 보존 특징)를 이해하고, 전문 LNK 파싱 도구(LECmd 등)를 활용하여 삭제된 프로그램의 원시 속성 시간을 도출합니다.

## 4. 의도한 풀이 흐름
1. **LNK 파일 메타데이터 파싱**:
   * 제공된 `evidence.lnk` 파일을 분석하기 위해 Eric Zimmerman의 `LECmd.exe` 유틸리티를 실행합니다:
     `LECmd.exe -f evidence.lnk`
   * 또는 리눅스 터미널의 파이썬 기반 LNK 파서 도구를 가동합니다.
2. **타깃 파일 타임스탬프 감사**:
   * 파싱 결과 출력 정보 중 **Target Created (또는 Target Creation Time)** 속성을 식별합니다:
     * **바로가기 파일 생성 시각 (LNK Created)**: `2026-06-28 11:15:30`
     * **대상 타깃 파일 생성 시각 (Target Created)**: `2026-06-28 10:45:15`
     * **대상 타깃 파일 수정 시각 (Target Modified)**: `2026-06-28 10:45:15`
     *(주의: LNK 파일의 물리적 생성 일시가 아닌, 그 내부에 저장된 "대상(Target) 파일의 생성 일시" 정보를 추출해야 합니다)*
3. **최종 플래그 구성**:
   * 획득한 타깃 원본 생성 시각 `2026-06-28 10:45:15` (UTC 기점) 데이터를 요구 포맷 규격(YYYY-MM-DD_HH:MM:SS)으로 치환하여 최종 플래그를 도출합니다:
     `picoCTF{2026-06-28_10:45:15}`

## 5. 정답(플래그) 규칙 및 예시
* **플래그 형식**: `picoCTF{YYYY-MM-DD_HH:MM:SS}`
* **예시**: `picoCTF{2026-06-28_10:45:15}` (국제 분석 규격 준수를 위해 UTC 타임스탬프 기준 변환 결과로 통일하여 대입합니다)

## 6. 출제자 노트 & 파일 제작 가이드
* **아티팩트 제작 방법**:
  1. Windows 테스트 환경에서 로컬 시스템의 시각을 `2026-06-28 10:45:15 UTC`에 정렬합니다.
  2. 기밀 폴더 하위에 텍스트 파일이나 임의 바이너리 `attack_tool.exe`를 작성 생성합니다. (이 시점에 파일의 파일시스템 Creation Time 속성이 해당 시간으로 인가됩니다)
  3. 시스템 시각을 임의의 시간(예: `2026-06-28 11:15:30 UTC`)으로 변강한 뒤, 윈도우 바탕화면에서 해당 `attack_tool.exe` 바로가기 파일을 작성하거나 실행 행위를 수행해 바로가기를 강제 생성 유도합니다. (이때 생성되는 `evidence.lnk` 헤더 내 Target Creation Time에 `2026-06-28 10:45:15`가 정확히 박제됩니다)
  4. 원본 `attack_tool.exe` 파일을 휴지통에서도 비우고 파일 파쇄기로 영구 삭제합니다.
  5. 남아있던 `evidence.lnk` 파일을 덤프 획득하여 배포 아티팩트로 지정합니다.
* **출제 포인트**: 
  * 디바이스 내 공격 실행 파일이 파쇄 삭제(Wiped Executable)되어 물리적 메타데이터 링크가 분실되었더라도, 해당 파일을 가동할 때 OS에 의해 남는 LNK 메타데이터 헤더 속의 타깃 타임스탬프 보존 현상(LNK Target Time Metadata Forensics)을 역파싱하여 침입자가 최초 도구를 이식/빌드했던 정확한 시각을 증명하는 디지털 과학 수사 역량을 다집니다.

## 7. 트러블슈팅 및 힌트
* **Q. 바로가기 파일의 Creation Time과 Target Creation Time의 차이는 무엇인가요?**
  * A. `LNK Creation Time`은 단축 바로가기 파일(`.lnk`) 그 자체가 디스크 볼륨 상에 작성되어 적재된 시점을 나타냅니다. 반면, `Target Creation Time`은 해당 바로가기가 지목하여 연결 및 링크하고 있는 원본 대상 파일(Target File)이 가졌던 물리 파일시스템 상의 최초 생성 타임스탬프를 LNK 파일 바디 데이터 세그먼트에 복제 보존해 둔 것이므로 엄연히 다른 속성입니다.
* **Q. 타깃 타임스탬프가 변조(타임스톰핑)될 수도 있나요?**
  * A. 공격자가 디스크 상의 원본 실행 파일에 대해 타임스톰핑을 가한 뒤 바로가기 LNK가 생성되거나 업데이트되었다면, LNK 내부의 Target Creation Time에도 변조된 시간이 복사 기록될 수 있습니다. 하지만 LNK 파일이 작성된 이후에 이루어지는 원본 파일의 속성 변조는 이미 생성된 LNK의 Target Creation Time 헤더에는 소급 반영되지 않으므로, 공격자가 원본 파일 타임스톰핑 후 수정을 누락했거나, LNK 생성 시점 이후에 수정한 경우 시간 차이를 교차 비교하여 조작 여부를 입증해 내는 결정적 단서가 됩니다.

## 8. 학습 포인트
* **LNK 단축 링크 바이너리 스펙**: ShellLinkHeader 구조체 오프셋 배치 정보 및 타깃 타임스탬프 저장 바이트 명세를 상세 습득합니다.
* **삭제 바이너리 속성 역추적**: 지워진 실행 파일에 대한 역사적 생성 흔적을 LNK Target Timestamps 분석을 통해 논리 복구하는 과학 수사적 연계 능력을 배양합니다.
