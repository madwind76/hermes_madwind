# 파이썬 컴파일 바이너리 내 컴파일러 파일 사이즈 크기 규명 (Python pyc File Size Verification)

> **난이도**: 초급  
> **소요 시간**: 15~20분  
> **참고 picoCTF 문제**: 파이썬 컴파일 바이너리 원본 파일 크기 추출 (Python .pyc Header Size 분석)

## 1. 배경 시나리오
서버 침투 조사의 핵심 증적으로 회수된 백도어 프로그램인 `agent_check.pyc` 파일이 확보되었습니다. 피의자는 수사관들이 파일의 원형을 확인하기 어렵게 하고 분석 자동화 도구의 무결성 검증을 교란하기 위해, 컴파일된 바이트코드 바이너리 끝부분에 대량의 더미 쓰레기 주석 바이트를 인위적으로 패딩하여 파일 크기를 크게 부풀려 둔 상태입니다. 하지만 파이썬 컴파일 라이브러리는 빌드 시 헤더의 특정 오프셋 영역에 **컴파일되기 직전 원본 소스 코드 파일(`.py`)의 순수 물리적 바이트 크기 정보(Original Source File Size)**를 4바이트 정수 값으로 박제하여 보존합니다. 수집한 `agent_check.pyc` 파일의 바이너리 헤더를 정적 점검하여, **컴파일 당시 원본 파이썬 소스 코드 파일의 정확한 크기(플래그)**를 바이트 단위 정수로 도출하십시오.

## 2. 제공 파일
* `agent_check.pyc` (파이썬 3 환경에서 빌드되어 더미 바이트가 수동 패딩된 컴파일 바이너리 파일)

## 3. 문제 목표
파이썬 컴파일 `.pyc` 바이너리 헤더의 16바이트 메타데이터 레이아웃 구조(오프셋 12~15 위치에 할당되는 4바이트 리틀엔디언 원본 소스 파일 크기 필드의 기록 스펙)를 이해하고, 이를 정수 바이트 값으로 복원합니다.

## 4. 의도한 풀이 흐름
1. **헤더 데이터 확인**:
   * 제공된 `agent_check.pyc` 파일을 헥스 에디터로 로드합니다.
   * 파이썬 3.7+ 표준 헤더 스키마에 따라 오프셋 구조를 나열해 봅니다:
     * `0x00 ~ 0x03`: 매직 넘버 (4바이트)
     * `0x04 ~ 0x07`: 빌드 비트 필드 플래그 (4바이트)
     * `0x08 ~ 0x0B`: 수정 시각 타임스탬프 (4바이트)
     * `0x0C` ~ `0x0F`: 원본 소스 파일 크기 (4바이트 리틀엔디언 정수)
2. **크기 메타데이터 바이트 카빙**:
   * 오프셋 `0x0C` 위치부터 `0x0F` 위치까지 기록된 4바이트 데이터를 획득합니다:
     `A4 03 00 00`
3. **리틀엔디언 정수 변환**:
   * 획득한 헥사 바이트를 리틀엔디언(바이트 스왑) 순서로 교정하여 16진수 값을 정립합니다:
     `A4 03 00 00` -> `0x000003A4`
   * 16진수 `0x3A4`를 10진수로 정수 변환합니다:
     `932` (바이트)
4. **플래그 도출**:
   * 계산된 정수 크기 결과물인 `932` 값을 정답 규격 포맷에 대입하여 최종 플래그를 정립합니다:
     `picoCTF{932_bytes_original_size}`

## 5. 정답(플래그) 규칙 및 예시
* **플래그 형식**: `picoCTF{<size_value>_bytes_original_size}`
* **예시**: `picoCTF{932_bytes_original_size}`

## 6. 출제자 노트 & 파일 제작 가이드
* **아티팩트 제작 방법**:
  1. 932바이트의 파일 크기를 지닌 원본 파이썬 스크립트 코드 `agent_check.py`를 작성합니다. (임의의 로직 코드 및 주석을 조합하여 정확히 932바이트로 크기를 맞춥니다)
  2. 파이썬 3 컴파일러를 구동하여 pyc 파일을 컴파일 빌드합니다:
     `python3 -m py_compile agent_check.py`
     (컴파일이 처리되면 빌드 엔진이 `agent_check.pyc` 헤더 오프셋 12 위치에 932의 리틀엔디언 표현식인 `A4 03 00 00` 바이트를 자동 기록합니다)
  3. 빌드된 pyc 파일의 최하단 종단부 뒤에 추가적인 대용량 더미 데이터나 알파벳 주석 코드를 수 메가바이트 크기로 덧붙여 파일 크기를 임의 조작합니다.
  4. 변조된 pyc 파일을 배포용 챌린지 아티팩트로 포장 제공합니다.
* **출제체 포인트**: 
  * 공격자가 바이너리 크기를 패딩 변조(File Size Padding Evasion)하여 해시 감지를 무력화하더라도, 컴파일 헤더 메타데이터 영역(pyc Size Metadata Forensics)에 온전히 잔재하는 원본 컴파일 전 소스 크기 속성을 헥사 수준에서 역연산해 내어 도구의 본질적 아이덴티티를 비교 입증하는 수사 기법을 훈련시킵니다.

## 7. 트러블슈팅 및 힌트
* **Q. 파이썬 컴파일러는 왜 원본 파일의 크기를 헤더에 기입해 놓나요?**
  * A. 파이썬 런타임 엔진이 `.pyc` 캐시 파일을 불러와 가동할 때, 현재 디스크 상의 원본 `.py` 파일의 실제 크기 및 수정 타임스탬프를 헤더 값과 실시간 상호 대조하기 위함입니다. 만약 원본 소스 파일의 크기나 시간이 헤더 정보와 어긋날 경우, 파이썬 인터프리터는 캐시 바이너리가 유효하지 않다고 판단하여 다시 컴파일하는 루틴을 밟게 되므로 내부 호환성 동기화 인증 메커니즘을 위해 기록 보존되는 필수 속성입니다.
* **Q. 리틀엔디언 바이트 스왑 연산이 헷갈립니다. 수식 연산은 어떻게 하나요?**
  * A. 파이썬 CLI에서 `struct` 모듈을 사용하면 리틀엔디언 바이트를 즉시 10진수 정수로 안전하게 디코딩할 수 있어 실수를 방지할 수 있습니다:
     ```python
     import struct
     raw_bytes = b'\xa4\x03\x00\x00'
     original_size = struct.unpack('<I', raw_bytes)[0]
     print(original_size) # 출력 결과: 932
     ```

## 8. 학습 포인트
* **Python 3 바이트코드 헤더 명세**: 컴파일 바이너리 파일 내부의 저수준 메모리 레이아웃 및 16바이트 구조체 정의 사양을 이해합니다.
* **에포크 타임(Epoch Time) 역연산**: 리틀엔디언 헥사 데이터를 32비트 정수로 계산하고, 이를 표준시(UTC) 기준 날짜 문자열로 복구해 포렌식 시간 증적을 확보하는 절차를 습득합니다.
