# PDF 인쇄 속의 위장 메타데이터 (PDF Printer Spooler Artifacts)

> **난이도**: 초중급  
> **소요 시간**: 25~30분  
> **참고 picoCTF 문제**: 윈도우 인쇄 스풀 흔적 분석 (Printer Spooler SPL/SHD 카빙 실무)

## 1. 배경 시나리오
사내 핵심 기밀 설계 문서가 종이 문서 형태로 무단 인쇄된 정황이 있습니다. 용의자는 인쇄 직후 물리적인 인쇄물은 파쇄하고 PC에서 원본 문서를 삭제했습니다. 그러나 윈도우 OS의 인쇄 메커니즘 상, 프린터로 전송되기 전 임시 보존되는 스풀 디렉터리(`C:\Windows\System32\spool\PRINTERS\`) 영역에서 인쇄 메타데이터를 담은 `00002.SHD` 파일과 실제 출력 내용을 담고 있는 `00002.SPL` 파일이 포렌식 복구되었습니다. 이 두 파일을 분석하여 **인쇄를 요청한 사용자 계정명**과 **출력물 내용 안에 삽입되어 있던 기밀 플래그**를 획득해야 합니다.

## 2. 제공 파일
* `00002.SHD` (프린터 스풀러 섀도 파일 - 인쇄 메타데이터 보존 파일)
* `00002.SPL` (프린터 스풀러 데이터 파일 - 실제 출력 바이너리 파일)

## 3. 문제 목표
윈도우 인쇄 스풀러의 기본 동작 구조 및 두 파일(SHD, SPL)의 역할을 이해하고, 섀도(SHD) 파일 파싱을 통해 인쇄 소유자를 복구하며, 스풀(SPL) 파일에서 프린터 드라이버 규격(RAW, EMF 또는 PDF)에 맞게 원본 문서를 추출/카빙하여 플래그를 복원합니다.

## 4. 의도한 풀이 흐름
1. **메타데이터 파싱 (SHD 분석)**:
   * 윈도우 SHD(Shadow) 파일 분석 도구(예: `SHD Parser` 또는 레지스트리/스풀 분석 스크립트)를 활용해 `00002.SHD` 파일을 파싱합니다.
   * 출력물 정보에서 인쇄 요청 계정명(예: `madmin`)과 인쇄 문서명(예: `draft_roadmap.pdf`)을 확인합니다.
2. **실제 출력 데이터 카빙 (SPL 분석)**:
   * `00002.SPL` 파일의 매직 바이트 시그니처를 확인하기 위해 헥스 에디터로 엽니다.
   * 드라이버 설정에 따라 PDF 기표 시그니처(`%PDF-`)가 시작 위치 또는 특정 오프셋에 존재하는지 확인합니다.
     *(만약 EMF 규격인 경우 `01 00 00 00 58 00 00 00` 등의 EMF 매직 바이트가 나타날 것입니다)*
3. **원본 포맷 복구**:
   * 본 시나리오에서는 SPL 내부에 원본 PDF 바이트가 온전히 스트리밍되어 있습니다. 오프셋 0부터 `%PDF-` 헤더가 존재하므로, 단순히 파일 확장자를 `.pdf`로 변경(`00002.SPL` -> `recovered.pdf`)하여 정상적인 문서로 복구합니다.
4. **기밀 플래그 획득**:
   * 복구된 `recovered.pdf` 문서를 열어 슬라이드/문서 하단 마진 또는 워터마크에 희미하게 기록된 플래그 문자열을 발견합니다.
     (예: `picoCTF{spool_file_carving_success}`)
5. **최종 플래그 작성**:
   * 사용자 계정명과 획득한 플래그 값을 결합합니다:
     `picoCTF{madmin_spool_file_carving_success}`

## 5. 정답(플래그) 규칙 및 예시
* **플래그 형식**: `picoCTF{<username>_<flag_string>}`
* **예시**: `picoCTF{madmin_spool_file_carving_success}`

## 6. 출제자 노트 & 파일 제작 가이드
* **아티팩트 제작 방법**:
  1. 기밀 플래그(`picoCTF{spool_file_carving_success}`)를 포함하는 1페이지짜리 문서를 PDF 형식으로 작성합니다.
  2. Windows 환경에서 임의의 PDF 프린터 드라이버를 통해 해당 문서를 인쇄합니다. 이때 프린터 속성 설정에서 "인쇄 작업 일시 중지"를 활성화합니다.
  3. 스풀러 관리 디렉터리 `C:\Windows\System32\spool\PRINTERS\` 로 이동하여 실시간 생성된 `*.SPL` 및 `*.SHD` 파일을 복제하여 확보합니다.
  4. 확보한 섀도 파일 내에 기재되는 계정명을 강제로 `madmin`으로 헥스 수정하거나, 모의 테스트 단계에서 해당 계정명 상태로 인쇄 작업을 생성해 배포용 아티팩트를 확정합니다.
* **출제 포인트**: 
  * 디스크 이미징이나 일반 파일 복구 범위 외에 잘 다뤄지지 않는 인쇄 서브시스템 아티팩트의 복구 기법을 실습하고, 윈도우 백그라운드 스풀러 구조의 증거력을 파악하게 합니다.

## 7. 트러블슈팅 및 힌트
* **Q. SPL 파일의 확장자를 pdf로 바꾸고 열었는데 파일 포맷 손상 에러가 발생합니다.**
  * A. 프린터 드라이버가 인쇄용 제어 언어(PCL, PostScript 등)를 추가 헤더로 덧붙였기 때문일 수 있습니다. 헥스 에디터로 파일을 열고 처음 등장하는 `%PDF-` 바이트 오프셋 이전의 더미 데이터 영역을 모두 지운(Truncate) 뒤 저장하여 다시 열어 보십시오.
* **Q. SHD 파서가 실행되지 않습니다.**
  * A. 윈도우 SHD 포맷 명세는 단순 구조체 형태로 구성되어 있습니다. 파서가 없다면 헥스 에디터 상에서 아스키(UTF-16LE 또는 ANSI) 디코딩 뷰를 켠 채 하단을 탐색하면 계정명 문자열(`madmin`)을 수동으로 식별할 수 있습니다.

## 8. 학습 포인트
* **윈도우 인쇄 메커니즘**: 인쇄 요청이 접수될 때 호스트 운영체제가 임시 큐(Queue)를 관리하기 위해 생성하는 SPL 및 SHD 파일의 저수준 명세 구조를 학습합니다.
* **바이너리 트런케이트(Truncation)**: 스트림 컨테이너 내부에 임베디드된 특정 문서 포맷 바이트 오프셋을 역추적하여 앞단 제어 헤더를 잘라내고 복원하는 실전 카빙 기술을 습득합니다.
