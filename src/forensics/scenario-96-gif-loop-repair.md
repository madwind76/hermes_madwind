# 변조된 GIF 헤더 및 루프 제어 블록 수동 복원 (Corrupted GIF Application Extension Repair)

> **난이도**: 초중급  
> **소요 시간**: 25~30분  
> **참고 picoCTF 문제**: 손상된 애니메이션 GIF 루프 메타데이터 복원 (GIF Application Extension 분석)

## 1. 배경 시나리오
공격 유출 정황이 담긴 파일인 `loop_corrupted.gif` 애니메이션 이미지 파일이 발견되었습니다. 이 파일은 첫 번째 프레임만 단색으로 정적으로 출력될 뿐, 다음 장으로 넘어가는 애니메이션 루프가 정상 동작하지 않거나 파일 구조 에러가 발생합니다. 정밀 분석 결과, 위협 행위자는 이미지 디코더가 애니메이션 추가 프레임들을 연속 렌더링하도록 정의하는 핵심 메타데이터 영역인 **애플리케이션 확장(Application Extension) 제어 블록**의 선두 3바이트를 고의로 0으로 소거하여 렌더러를 교란했습니다. GIF89a 표준 명세에 입각하여 8바이트의 식별자 `NETSCAPE`와 3바이트의 인증 코드 `2.0`으로 표기되는 루프 제어 영역을 수동 복구(Hex Patching)하고, **정상 작동하는 애니메이션 프레임 스퀀스 상에서 교차 출력되는 최종 플래그**를 획득하십시오.

## 2. 제공 파일
* `loop_corrupted.gif` (애플리케이션 확장 제어 블록인 NETSCAPE2.0 필드 정보가 인위적으로 훼손된 애니메이션 GIF 파일)

## 3. 문제 목표
GIF89a 규격 내의 확장 블록 유형인 **Application Extension Block (시작 바이트 `21 FF 0B`)** 구조와 애니메이션 프레임 루프 카운트를 정의하는 하위 세그먼트 메타데이터를 이해하고, 헥스 에디터로 깨진 바이너리를 복구합니다.

## 4. 의도한 풀이 흐름
1. **바이너리 영역 헥스 검색**:
   * 제공된 `loop_corrupted.gif` 파일을 헥스 에디터로 엽니다.
   * 아스키 텍스트 영역에서 애니메이션 루프를 담당하는 사양 식별 문자열인 **`NETSCAPE2.0`** (헥사: `4E 45 54 53 43 41 50 45 32 2E 30`) 검색을 기동합니다.
   * 파일 오프셋 `0x320` 부근에서 검색 매칭된 문자열을 확보하고 바로 앞의 3바이트 구조를 대조합니다:
     * **현재 바이트 상태**: `00 00 00` `4E 45 54 53 43 41 50 45 32 2E 30` (즉, `NETSCAPE2.0` 앞이 영으로 손상됨)
2. **애플리케이션 확장 헤더 패칭 복원**:
   * GIF89a 포맷에서 외부 렌더링 옵션을 지시하는 애플리케이션 확장 블록은 다음 사양을 고수해야 합니다:
     * **Extension Introducer (1바이트)**: `0x21` (확장 마커)
     * **Application Extension Label (1바이트)**: `0xFF` (애플리케이션 제어 지정)
     * **Block Size (1바이트)**: `0x0B` (뒤이을 식별 문자열 길이인 11바이트 정의)
   * 따라서 오염되어 있던 `00 00 00` 영역을 표준 헤더인 **`21 FF 0B`** 로 덮어쓰기 패칭합니다.
   * 패치된 바이너리를 `fixed_loop.gif` 이름으로 저장합니다.
3. **애니메이션 재생 및 플래그 획득**:
   * 복구 완료한 `fixed_loop.gif` 파일을 브라우저나 애니메이션 지원 이미지 뷰어로 구동합니다.
   * 프레임이 루프 기동하며 첫 프레임 이후 뒤따르던 가려진 플래그 이미지 페이지가 교차 기동하여 최종 플래그를 노출합니다:
     `picoCTF{gif_app_ext_loop_netscape_rebuilt}`

## 5. 정답(플래그) 규칙 및 예시
* **플래그 형식**: `picoCTF{<flag_string>}`
* **예시**: `picoCTF{gif_app_ext_loop_netscape_rebuilt}`

## 6. 출제자 노트 & 파일 제작 가이드
* **아티팩트 제작 방법**:
  1. 1번 프레임에는 더미 가짜 이미지, 2번 프레임에는 기밀 플래그 `picoCTF{gif_app_ext_loop_netscape_rebuilt}` 이미지를 레이아웃 배치한 뒤 무한 루프 애니메이션 GIF로 기동 빌드합니다. (이때의 바이너리는 21 FF 0B NETSCAPE2.0 확장 블록을 내장하게 됩니다)
  2. 헥스 에디터로 컴파일된 GIF를 로드하여 `NETSCAPE2.0` 문자열 오프셋 바로 앞 3바이트인 `21 FF 0B`를 `00 00 00`으로 덮어써 루프 렌더러 작동 조건을 오염시킵니다.
  3. 변조 처리한 파일을 `loop_corrupted.gif`로 최종 저장해 수검 파일로 배포합니다.
* **출제 포인트**: 
  * 멀티 프레임 그래픽 명세(GIF89a Application Extension Specifications)의 작동 제어 블록 구조를 이해하고, 확장 블록 도입자 매직 바이트(`21 FF 0B`) 복구를 통해 가려진 추가 미디어 데이터 스트림을 가시화해 증적을 안전하게 구출해 내는 정밀 파일 수리 분석 역량을 검증합니다.

## 7. 트러블슈팅 및 힌트
* **Q. NETSCAPE2.0 블록 바로 뒤에 뒤따르는 데이터 서브 블록 구조는 어떻게 되어 있나요?**
  * A. 애플리케이션 인증 문자열인 11바이트 `NETSCAPE2.0` 바로 뒤에는 루프 옵션을 지정하는 3바이트 크기의 데이터 서브 블록과 종단 바이트가 뒤따릅니다:
    * `03` (데이터 서브 블록 크기 지시: 3바이트)
    * `01` (루프 제어 인덱스 값)
    * `00 00` (반복 횟수를 뜻하는 2바이트 리틀엔디언 정수: `00 00`은 무한 루프 재생을 의미)
    * `00` (블록의 물리 종단을 알리는 Terminator 바이트)
    본 문제에서는 선두 확장 헤더 3바이트만 교정하면 리더가 이 서브 블록 사양을 정상 독취하여 무한 루프 애니메이션을 재개하도록 설계되어 있습니다.
* **Q. 이미지 수정 후에도 루프가 돌지 않고 정지되어 있습니다.**
  * A. 기본 윈도우 정적 사진 뷰어 중 일부는 다중 프레임 GIF 루프 애니메이션 재생 기능을 기본 지원하지 않아 첫 프레임만 계속 보여줄 수 있습니다. 이 경우 크롬, 파이어폭스 등 현대식 웹 브라우저 창으로 복구된 GIF 파일을 끌어다 놓기(Drag & Drop)하여 기동하면 프레임 동적 루프 효과를 확실하게 확인할 수 있습니다.

## 8. 학습 포인트
* **GIF89a 확장 블록 아키텍처**: 그래픽 제어 및 애플리케이션 고유 확장 블록(NETSCAPE2.0)의 바이트 구조를 상세 학습합니다.
* **미디어 카빙 및 패칭**: 렌더링 무효화 우회를 깨부수기 위해 이진 메타데이터 블록을 교정 수리하고 숨겨진 타깃 데이터 스트림을 시각적으로 환원하는 기법을 마스터합니다.
