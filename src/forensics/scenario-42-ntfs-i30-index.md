# NTFS 인덱스 레코드 카빙 (NTFS $I30 Index Root/Allocation Carving)

> **난이도**: 초중급  
> **소요 시간**: 25~30분  
> **참고 picoCTF 문제**: 디렉터리 내 흔적 복구 및 MFT 바이너리 분석 (NTFS $I30 INDX 슬랙 영역 스캔)

## 1. 배경 시나리오
피의자는 컴퓨터 검수 직전 외부 유출용으로 제작했던 중요 파일을 삭제하고 해당 디스크 볼륨의 마스터 파일 테이블(MFT) 레코드까지 조각모음 및 덮어쓰기하여 파일 메타데이터 흔적을 완전히 소거했습니다. 하지만 NTFS 파일시스템은 특정 디렉터리 내에 어떤 파일들이 담겨 있었는지 인덱스 트리 형태로 관리하기 위해 `$I30` 인덱스 루트(Index Root) 및 인덱스 할당(Index Allocation) 속성을 활용합니다. 파일이 논리적으로 삭제되어도 이 `$I30` 인덱스 블록(INDX)의 슬랙(Slack) 공간에 삭제되기 전 존재했던 **원래 파일의 이름과 메타데이터 정보**가 고스란히 남아 있을 수 있습니다. 제공된 인덱스 블록 덤프 `i30_dump.bin` 파일을 정밀 파싱하여 **삭제되었던 파일의 명칭 내에 들어 있는 플래그**를 획득해야 합니다.

## 2. 제공 파일
* `i30_dump.bin` (대상 디렉터리의 NTFS `$I30` Index Allocation 스트림 영역에서 덤프한 4096바이트 크기의 원시 바이너리 파일)

## 3. 문제 목표
NTFS 파일시스템의 디렉터리 구조화 원리와 `$I30` 인덱스 엔트리 포맷(인덱스 레코드 시그니처 `INDX`, 엔트리 헤더, UTF-16LE 파일명 영역)을 분석하고, B-Tree 재조정 과정에서 슬랙(미할당) 영역으로 밀려나 삭제 처리된 파일 엔트리를 역추적해 냅니다.

## 4. 의도한 풀이 흐름
1. **바이너리 구조 분석**:
   * 제공된 `i30_dump.bin` 파일을 헥스 에디터로 엽니다.
   * 첫 4바이트 매직 시그니처가 `49 4E 44 58` (아스키 `INDX`)임을 검증하여 해당 파일이 NTFS 디렉터리 인덱스 할당 데이터 섹터임을 확정합니다.
2. **슬랙 영역 탐색 및 파일명 스캔**:
   * 디렉터리 인덱스 블록(일반적으로 4096바이트)은 유효 엔트리들이 가용 리스트 형태로 보존되고, 리스트 끝 마커(End of entries) 이후부터 블록 끝까지가 슬랙 공간으로 정의됩니다.
   * 지워진 파일명은 UTF-16LE(2바이트 유니코드) 형태로 잔재하므로, 바이트 사이사이에 `00`이 섞인 문자열 상태로 슬랙 공간에 남아 있게 됩니다.
   * 헥스 에디터 상에서 `picoCTF` 또는 유니코드 스트링을 스캔합니다:
     ```bash
     strings -e l i30_dump.bin | grep "picoCTF"
     ```
3. **인덱스 엔트리 정상 추출**:
   * 검색 결과 슬랙 영역 내부에서 다음 형식의 유니코드 파일명 문자열을 검출합니다:
     `flag_picoCTF{i30_ndx_slack_space_carving}.txt`
   * 파일명에 기록되어 있던 최종 플래그 값을 복원합니다:
     `picoCTF{i30_ndx_slack_space_carving}`

## 5. 정답(플래그) 규칙 및 예시
* **플래그 형식**: `picoCTF{<flag_string>}`
* **예시**: `picoCTF{i30_ndx_slack_space_carving}`

## 6. 출제자 노트 & 파일 제작 가이드
* **아티팩트 제작 방법**:
  1. Windows 가상 머신의 NTFS 볼륨에서 임의의 폴더를 생성합니다.
  2. 해당 폴더 내에 정상 파일 여러 개와 함께 `flag_picoCTF{i30_ndx_slack_space_carving}.txt` 파일을 생성합니다.
  3. 윈도우 탐색기에서 해당 플래그 파일을 삭제합니다. 그 후 인덱스 트리 재정렬을 유도하기 위해 다른 이름의 임시 파일들을 소량 추가/삭제합니다. (이때 B-Tree 노드가 재조정되며 구버전 엔트리가 슬랙 영역으로 강제 격리 보존됩니다)
  4. FTK Imager 등의 포렌식 도구를 이용하여 해당 디렉터리의 MFT 레코드로 이동, `$INDEX_ALLOCATION` (속성 ID: `$I30`) 스트림 영역을 덤프하여 `i30_dump.bin`으로 저장 및 배포합니다.
* **출제 포인트**: 
  * 휴지통이나 일반 파일 삭제(MFT 레코드 덮어쓰기) 수준을 넘어서서, 파일이 담겨 있던 폴더의 인덱스 물리 아티팩트($I30 INDX Slack)를 파싱하여 이전에 실존했던 파일 명칭과 생성 일시 흔적을 복구해 내는 고도의 디스크 카빙 역량을 기릅니다.

## 7. 트러블슈팅 및 힌트
* **Q. strings 명령어 출력 결과에 파일명이 나오지 않습니다.**
  * A. NTFS는 파일명을 **UTF-16LE(리틀 엔디언 유니코드)**로만 기록합니다. 일반 리눅스 `strings` 명령은 1바이트 ASCII 문자열만 탐색하므로 검색되지 않습니다. 반드시 인코딩 옵션 `-e l` (16-bit little-endian)을 주어 유니코드 스트링 스캔을 지시해야 정상 출력됩니다.
* **Q. INDX 레코드 내부에서 지워진 엔트리의 생성 시각 정보도 알 수 있나요?**
  * A. 네, `$I30` 인덱스 엔트리 내부에는 파일 정보 외에 해당 파일의 **MFT 생성/수정/액세스 시간(Filetime 8바이트 정수형 4개)** 구조체가 포함되어 있으므로, 헥스 오프셋 연산을 통해 삭제된 파일의 생성 및 삭제 직전 수정 타임라인을 정밀 입증할 수 있습니다.

## 8. 학습 포인트
* **NTFS 인덱스 아티팩트**: 폴더 내 파일 색인을 위해 B-Tree 구조로 제어되는 `$INDEX_ROOT`와 `$INDEX_ALLOCATION` 구조체 명세 및 레코드 배치 기법을 학습합니다.
* **디렉터리 슬랙 포렌식**: 데이터베이스나 파일시스템의 인덱스 리팩토링 과정에서 잔재하는 구버전 메타데이터 엔트리의 물리 잔재를 안전하게 식별해 복구해 내는 포렌식 논리를 터득합니다.
