# 윈도우 휴지통의 증언 (Windows Recycle Bin Forensics)

> **난이도**: 초중급  
> **소요 시간**: 25~30분  
> **참고 picoCTF 문제**: 삭제된 윈도우 파일 복구 및 메타 분석 (Recycle Bin $I/$R 아티팩트 파싱)

## 1. 배경 시나리오
보안 사고 의심자의 업무용 PC를 격리 분석하던 중, 바탕화면에서 중요 자료가 일괄 삭제된 기록이 발견되었습니다. 용의자는 해당 파일들을 완벽하게 소거했다고 주장했으나, 윈도우 파일 삭제 보존 메커니즘을 확인해 본 결과 휴지통 폴더 하위 디렉터리(`C:\$Recycle.Bin\<SID>\`)에 삭제된 원래 정보의 복구 키가 되는 인덱스 파일 `$I10243.txt`와 물리 본문 데이터인 `$R10243.txt`가 쌍으로 남아 있었습니다. 분석가는 인덱스 파일을 분석해 **삭제되기 전 원래 파일의 명칭**과 **물리 파일 내부의 은닉 플래그**를 알아내야 합니다.

## 2. 제공 파일
* `$I10243.txt` (휴지통 메타데이터 인덱스 파일)
* `$R10243.txt` (휴지통 본문 파일 - 물리 데이터 보존 파일)

## 3. 문제 목표
윈도우 비스타/7 이후부터 도입된 휴지통 구조의 핵심인 `$I` (Index) 파일의 바이너리 구조(헤더, 오프셋별 파일 크기, 삭제 시간, 파일명 영역)를 파악하고, `$R` (Resource) 본문 파일과 연계해 원래 파일명 정보와 원본 데이터를 온전히 추출해 냅니다.

## 4. 의도한 풀이 흐름
1. **아티팩트 구조 이해**:
   * 사용자가 윈도우에서 파일을 삭제하면 휴지통 디렉터리로 이동하며 파일명이 `$R<임의의문자>.<확장자>`로 바뀌어 원본 내용이 복사됩니다.
   * 이와 동시에, 원래 저장되어 있던 경로와 파일 크기, 삭제 시각 메타데이터가 담긴 `$I<임의의문자>.<확장자>` 인덱스 파일이 생성됩니다.
2. **인덱스 파일 파싱 ($I 분석)**:
   * **도구 활용**: Eric Zimmerman의 `RBCmd.exe` 또는 파이썬 `$I` 파서 스크립트를 사용해 `$I10243.txt` 파일을 해독합니다.
   * **수동 헥스 분석**:
     * `$I` 파일은 고정 헤더로 시작합니다.
     * 오프셋 8: 원래 파일 크기 (8바이트 정수)
     * 오프셋 16: 삭제 시간 (8바이트 Windows Filetime 형식)
     * 오프셋 24: 원래 파일의 전체 절대 경로 및 파일명 (UTF-16LE 문자열)
   * 파싱 결과 원래 파일명이 `secret_flag.txt` 이었음을 탐지합니다.
3. **원본 파일 데이터 획득 ($R 분석)**:
   * 본문에 해당하는 `$R10243.txt` 파일을 텍스트 에디터나 카빙을 통해 열어 봅니다.
   * 내부 본문 문자열을 획득합니다:
     `picoCTF{recycle_bin_h0lds_deleted_secrets}`
4. **최종 플래그 조합**:
   * 원래 파일명(확장자 포함)과 획득한 플래그 문자열을 언더스코어로 결합해 플래그를 입력합니다:
     `picoCTF{secret_flag.txt_recycle_bin_h0lds_deleted_secrets}`

## 5. 정답(플래그) 규칙 및 예시
* **플래그 형식**: `picoCTF{<original_filename>_<flag_string>}`
* **예시**: `picoCTF{secret_flag.txt_recycle_bin_h0lds_deleted_secrets}`

## 6. 출제자 노트 & 파일 제작 가이드
* **아티팩트 제작 방법**:
  1. 기밀 파일 `secret_flag.txt`를 생성하고 내용으로 `picoCTF{recycle_bin_h0lds_deleted_secrets}`를 입력합니다.
  2. 윈도우 탐색기에서 해당 파일을 삭제(`Del`키 사용, 휴지통으로 이동)합니다.
  3. 볼륨의 숨김 해제 후 `$Recycle.Bin` 디렉터리 아래의 해당 사용자 SID 폴더 내에 들어가 방금 생성된 `$I`로 시작하는 파일과 `$R`로 시작하는 한 쌍의 파일을 추출합니다.
  4. 테스트 중 파일의 확장자가 변경되지 않도록 `$I10243.txt` 형태로 파일명을 치환하고, 인덱스 파일 내의 텍스트가 정상적으로 파싱되는지 역검증한 후 배포합니다.
* **출제 포인트**: 
  * 윈도우 휴지통 아티팩트의 `$I` 와 `$R` 한 쌍의 상호 결합 관계(Recycle Bin Forensics)를 규명하고, 파일이 논리적으로 삭제되어 원래 명칭을 잃었을 때 메타데이터 구조를 이용해 사건을 복원하는 절차를 습득시킵니다.

## 7. 트러블슈팅 및 힌트
* **Q. 인덱스 파일 내용이 헥스 뷰어에서 깨져 보입니다.**
  * A. 원래 파일 경로 명세 영역은 일반 아스키가 아닌 **UTF-16LE(Unicode)** 문자열로 기록되므로 바이트 사이사이에 `00`이 섞여 나타납니다. 헥스 에디터의 인코딩 보기 방식을 유니코드로 변경하면 정상적으로 경로가 가독됩니다.
* **Q. Windows Filetime 시각 변환은 어떻게 하나요?**
  * A. 오프셋 16 위치의 8바이트 데이터는 1601년 1월 1일 이후의 100나노초 간격을 센 64비트 정수입니다. Dcode와 같은 시간 변환 도구에 해당 헥스 바이트를 입력하면 정확한 인간 가독 시각으로 매핑됩니다.

## 8. 학습 포인트
* **윈도우 휴지통(Recycle Bin) 포렌식**: 휴지통의 물리적 구조 및 인덱싱 정책을 파악하여 사용자가 고의로 삭제한 데이터의 명칭 복구 가능 여부를 검증합니다.
* **Windows Filetime 스키마**: 윈도우 운영체제 레벨에서 타임스탬프를 직렬화하는 64비트 마이크로/나노초 에포크 시간 계산법을 학습합니다.
