# 변조된 PNG IDAT 청크 압축 해제 오류 복원 (PNG IDAT zlib Repair)

> **난이도**: 초중급  
> **소요 시간**: 25~30분  
> **참고 picoCTF 문제**: 손상된 이미지 데이터 압축 스트림 복원 (PNG IDAT zlib 포렌식)

## 1. 배경 시나리오
압수수색 단계에서 기밀 유출 정황이 담긴 파일 `corrupted_idat.png`를 획득했습니다. 파일 시그니처(`\x89PNG`) 및 시작 헤더인 `IHDR` 청크는 이상이 없으나, 파일을 더블 클릭하면 "이미지 압축 해제 오류" 또는 "정의되지 않은 이미지 본문 규격" 메시지가 나타나며 뷰어에 회색 화면만 렌더링되는 안티 포렌식 행위가 가해졌습니다. 분석가들은 픽셀 이미지 데이터 본문을 보유하는 **IDAT 청크**의 압축 알고리즘 데이터 영역 시작부에 가벼운 오염이 존재함을 확인했습니다. IDAT가 차용하는 **zlib 압축 스트림(RFC 1950)**의 헤더 바이트 규격을 찾아 복원함으로써 이미지를 정상 디코딩하고 **본문 안의 플래그**를 획득하십시오.

## 2. 제공 파일
* `corrupted_idat.png` (IDAT 청크 내 zlib 압축 시작 2바이트 영역이 오염되어 압축 풀기가 실패하는 PNG 파일)

## 3. 문제 목표
PNG 파일의 픽셀 이미지 본문 영역인 `IDAT` 청크 구조와 해당 청크 데이터가 차용하는 Deflate/zlib 압축 명세(RFC 1950 zlib Header 규격: CMF 및 FLG 바이트 데이터 구성)를 파악하고, 헥스 에디터를 이용해 고의 변조된 헤더 2바이트를 복원하여 압축 알고리즘을 복원해 냅니다.

## 4. 의도한 풀이 흐름
1. **바이너리 정밀 정적 분석**:
   * 제공된 `corrupted_idat.png` 파일을 헥스 에디터로 엽니다.
   * `IHDR` 청크는 정상이고, 이어서 실제 픽셀 비트맵 정보가 압축 적재되는 **IDAT** 청크의 위치를 검색합니다.
   * 아스키 텍스트 영역에서 `IDAT` 문자열(헥스 값: `49 44 41 54`)을 색인합니다.
2. **zlib 스트림 시작 오프셋 진단**:
   * PNG 명세 상, `IDAT` 식별 문자열 4바이트 바로 다음(오프셋 +4) 위치부터 실제 zlib 디플레어(Deflate) 압축 알고리즘이 적용된 이진 데이터 스트림이 출발합니다.
   * 현재 `IDAT` 문자열 직후 2바이트 공간의 데이터 값을 검사해 보니 `00 00` (또는 `FF FF` 등 깨진 바이트)으로 훼손되어 있습니다.
3. **zlib RFC 1950 스펙 대조 및 수정**:
   * zlib 사양상, 기본 압축(Deflate) 알고리즘에 32KB 윈도우 크기를 사용하고 기본 압축률 레벨(Level 6)로 기동된 기본 스트림의 첫 2바이트(CMF - Compression Method and Flags / FLG - Flags)는 항상 **`78 9C`** 로 수립되어야 함을 알아냅니다 (혹은 최상위 압축률의 경우 `78 DA`).
   * 헥스 에디터 상에서 `IDAT` (`49 44 41 54`) 문자열 바로 다음에 등장하는 깨진 2바이트 값을 원래 정규 사양인 **`78 9C`** 로 덮어쓰기 패치합니다.
   * 수정본을 `repaired_idat.png`로 내보냅니다.
4. **이미지 디코딩 및 플래그 획득**:
   * `repaired_idat.png`를 렌더링합니다. zlib 라이브러리가 에러 없이 본문 압축을 정상적으로 디플레이션(Decompression)하여 이미지가 출력되고, 그 안에 그려진 플래그를 색출해 냅니다:
     `picoCTF{zlib_id4t_defl4te_header_fixed}`

## 5. 정답(플래그) 규칙 및 예시
* **플래그 형식**: `picoCTF{<flag_string>}`
* **예시**: `picoCTF{zlib_id4t_defl4te_header_fixed}`

## 6. 출제자 노트 & 파일 제작 가이드
* **아티팩트 제작 방법**:
  1. 기밀 플래그 `picoCTF{zlib_id4t_defl4te_header_fixed}` 문자열을 페인트 툴로 드로잉하여 PNG 이미지로 빌드합니다.
  2. 헥스 에디터로 이미지 파일을 불러옵니다.
  3. `IDAT` 문자열(`49 44 41 54`) 직후 오프셋에 적힌 zlib 매직 헤더 `78 9C` 바이트를 임의의 더미 바이트 `00 00`으로 덮어씁니다.
  4. 수정한 파일을 `corrupted_idat.png`로 저장하고 이를 사용자에게 배포합니다.
* **출제 포인트**: 
  * 파일 포맷의 껍데기 헤더(시그니처) 검증 외에, 이미지 가독 렌더러가 내부 zlib 데이터 디플레이션을 기동할 때 필수로 요구하는 내부 라이브러리 규격(zlib Stream Specifications)의 세부 명세 결함을 복원해 내는 데이터 카빙 능력을 기릅니다.

## 7. 트러블슈팅 및 힌트
* **Q. zlib 헤더가 78 9C 외에 다른 바이트를 갖는 경우도 존재하나요?**
  * A. 네, 그렇습니다. CMF와 FLG 구조 설계에 따라 달라지며, 주로 사용되는 zlib 매직 헤더 정보는 다음과 같습니다:
    * `78 01`: 압축률 없음 (No Compression / Low)
    * `78 9C`: 기본 압축률 (Default Compression)
    * `78 DA`: 최대 압축률 (Maximum Compression)
    이 바이트들은 FCHECK 비트 체크섬 조건(두 바이트 값을 하나의 16비트 정수로 해석했을 때 31의 배수여야 함)을 항상 충족해야 라이브러리가 오류를 내지 않습니다.
* **Q. pngcheck 유틸리티가 이 오류도 찾아낼 수 있나요?**
  * A. 네, `pngcheck` 도구를 기동하면 다음과 같이 구체적인 실패 부위와 에러를 정적으로 출력해 줍니다:
     `corrupted_idat.png  zlib error: inflateInit() failed (-2)`
     이 출력을 통해 IDAT 청크의 zlib 압축 시작 헤더에 오류가 발생해 라이브러리 초기화(`inflateInit`)가 실패했음을 인지하고 즉각적인 Hex 교정을 진행할 수 있습니다.

## 8. 학습 포인트
* **PNG IDAT 구조 및 zlib 사양**: 이미지 본문 픽셀 데이터가 Deflate로 래핑되어 IDAT 청크 구조체에 직렬화 분할 적재되는 스키마를 이해합니다.
* **zlib 압축 라이브러리 복구**: RFC 1950 규격 명세에 의거해 CMF 및 FLG 검증 체크섬 관계를 진증하고, 손상된 바이트 패치를 통해 이미지 전체의 복원 프로세스를 완벽히 수립합니다.
