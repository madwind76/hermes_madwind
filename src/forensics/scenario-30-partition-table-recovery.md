# 삭제된 파티션 테이블 복구 (Partition Table Recovery)

> **난이도**: 초중급  
> **소요 시간**: 25~30분  
> **참고 picoCTF 문제**: 디스크 파괴 안티 포렌식 무력화 (MBR 파티션 복구 및 VBR 스캐닝 기법)

## 1. 배경 시나리오
압수수색 영장 집행 직전, 용의자는 외장 백업 하드디스크의 데이터를 복구할 수 없도록 파괴하기 위해 디스크 관리 도구를 악용하여 파티션 정보 전체를 삭제하고 디스크 전체가 초기화(Unallocated)된 것처럼 위장했습니다. 분석가가 수집한 원시 디스크 이미지 `destroyed_partition.img`를 분석 도구로 열어보려 시도했으나 "유효한 파티션 테이블이 존재하지 않음" 에러가 뜹니다. 용의자는 디스크 첫 섹터(LBA 0)의 MBR(Master Boot Record) 파티션 영역(오프셋 446~509)을 0으로 밀어버린 것으로 의심됩니다. 디스크 바이트 영역을 수동 스캔하여 유실된 원래 파티션의 위치를 찾고 기밀 플래그를 카빙해야 합니다.

## 2. 제공 파일
* `destroyed_partition.img` (MBR 파티션 테이블 영역이 Zero-out 처리된 100MB 크기의 Raw 디스크 이미지 파일)

## 3. 문제 목표
디스크 볼륨 레이아웃 구조(MBR, GPT, VBR)를 이해하고, MBR 파티션 테이블이 훼손되었을 때 파일시스템 고유 매직 시그니처를 스캔해 실제 데이터 볼륨이 시작되는 볼륨 부트 레코드(VBR)의 물리 섹터 주소를 특정하고, 이를 오프셋 파라미터로 지정하여 디스크 내부 파일을 안전하게 마운트 또는 카빙 복원합니다.

## 4. 의도한 풀이 흐름
1. **문제 상태 진단**:
   * `mmls destroyed_partition.img` 또는 `gdisk -l destroyed_partition.img` 명령어를 실행하면 파티션이 전혀 검출되지 않거나 에러가 납니다. LBA 0(MBR)의 파티션 구조 정보가 유실되었기 때문입니다.
2. **볼륨 시작 지점 스캔 (VBR 찾기)**:
   * 파일시스템이 실제 시작되는 VBR(Volume Boot Record)의 시그니처를 디스크 원시 바이트 상에서 고속 검색합니다.
   * 일반적으로 포맷된 NTFS 파티션 VBR의 시작 매직 바이트는 `EB 52 90 4E 54 46 53 20` (아스키 명세: `.R.NTFS ` 또는 `NTFS    `) 형태를 갖습니다.
   * `strings` 명령어와 바이트 오프셋 출력을 연동하거나, 파이썬 스크립트 또는 헥스 검색으로 `NTFS` 문자열을 탐색합니다:
     ```bash
     grep -a -b -o "NTFS    " destroyed_partition.img
     ```
   * 검색 결과 특정 물리 오프셋 위치(예: `1,048,576` 바이트 지점)에서 NTFS 시그니처가 시작됨을 발견합니다.
3. **물리 섹터 주소 환산**:
   * 1섹터는 512바이트이므로, 오프셋 바이트를 512로 나누어 섹터 번호를 역산합니다:
     `1,048,576 / 512 = 2048` 섹터
   * 즉, 실제 NTFS 데이터 파티션은 **2048 섹터** 지점부터 시작됨을 판단합니다.
4. **오프셋 지정을 통한 파일 시스템 복원**:
   * 슬루스킷 도구에 파티션 시작 섹터 번호(`-o 2048`)를 수동 기입하여 내부 파일 목록을 출력합니다:
     ```bash
     fls -r -o 2048 destroyed_partition.img
     ```
   * 내부 파일 트리 목록에서 기밀 파일인 `flag.txt` 및 그에 해당하는 Inode 번호(예: `56`)를 식별합니다.
   * `icat` 명령을 이용해 해당 파일 내용(플래그)을 카빙 추출합니다:
     `icat -o 2048 destroyed_partition.img 56`
5. **결과 확인**:
   * 추출한 기밀 플래그를 확인합니다:
     `picoCTF{mbr_partition_table_reconstructed}`

## 5. 정답(플래그) 규칙 및 예시
* **플래그 형식**: `picoCTF{<flag_string>}`
* **예시**: `picoCTF{mbr_partition_table_reconstructed}`

## 6. 출제자 노트 & 파일 제작 가이드
* **아티팩트 제작 방법**:
  1. 100MB 크기의 가상 디스크 이미지를 생성하고 파티션을 나눕니다:
     `dd if=/dev/zero of=destroyed_partition.img bs=1M count=100`
  2. 루프백 디바이스를 이용해 파티션을 NTFS로 포맷하고, 마운트하여 `flag.txt`에 `picoCTF{mbr_partition_table_reconstructed}`를 입력 및 저장합니다.
  3. 디스크 이미지 파일(`destroyed_partition.img`)의 첫 512바이트 영역(MBR) 중 파티션 엔트리 블록(오프셋 446부터 64바이트)의 값들을 헥스 에디터를 사용해 강제로 `00`으로 채워 지웁니다.
  4. 파티션 테이블 복구 툴(TestDisk 등)을 구동하거나, 수동 섹터 연산(`-o 2048`) 방식을 이용해 디스크가 정상 탐색 복구되는지 검증한 후 배포합니다.
* **출제 포인트**: 
  * 디스크 MBR 구조 파괴 수법에 맞서, 파일시스템 고유 시그니처 분석(VBR Signature Scanning)과 섹터 단위 변환 연산을 통해 저수준에서 파일시스템 레이아웃을 논리 복원해 내는 하드코어 디스크 포렌식 기법을 실습합니다.

## 7. 트러블슈팅 및 힌트
* **Q. mmls 명령어 결과만으로 파티션을 마운트할 수 없는 이유가 무엇인가요?**
  * A. 리눅스 `mount` 명령이나 `mmls` 도구는 디스크 LBA 0 영역의 파티션 테이블 정보(MBR 파티션 레코드)를 파싱해 위치를 정합니다. 이 영역이 손상되면 볼륨이 통째로 날아간 것처럼 착각하기 쉬우나, 실제 데이터가 들어 있는 파티션 물리 바이트 영역(LBA 2048 등)은 뒤에 고스란히 살아 있습니다.
* **Q. FAT32 파티션인 경우에는 VBR 시그니처가 무엇인가요?**
  * A. FAT32 파일시스템 VBR의 경우, 시작 바이트가 `EB 58 90`로 나타나며, 오프셋 82(0x52) 부근에 `FAT32   ` 명세 문자열이 아스키 형태로 기록되므로 이를 검색 패턴 키워드로 사용하면 동일한 역산이 가능합니다.

## 8. 학습 포인트
* **파티션 레이아웃(Partition Layout)**: MBR(Master Boot Record), GPT(GUID Partition Table)의 구조체 레이아웃 및 64바이트 파티션 레코드 포맷을 상세히 이해합니다.
* **VBR 시그니처 스캐닝**: 안티 포렌식 목적의 파티션 파괴를 우회하기 위해, 데이터 파티션 첫 머리(Volume Boot Record)의 물리 매직 바이트 시그니처를 스캔하고 분석 인자로 대입하는 저수준 볼륨 재구축 기법을 학습합니다.
