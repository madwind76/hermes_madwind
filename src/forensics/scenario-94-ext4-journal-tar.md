# ext4 저널링 기반 유출 데이터 압축 아카이브 TAR 복구 (ext4 Journal Tar Carving)

> **난이도**: 초중급  
> **소요 시간**: 25~30분  
> **참고 picoCTF 문제**: 삭제된 TAR 아카이브 물리 카빙 복구 (ext4 Journal Tar Carving)

## 1. 배경 시나리오
공격자가 웹 서버 내부의 여러 민감 설정 정보 파일들을 `exfil.tar` 파일로 다중 아카이빙하여 외부로 전송하고, 흔적을 지우기 위해 해당 tar 아카이브를 리눅스 터미널에서 `rm` 명령어로 영구 삭제했습니다. ext4 파일시스템 특성상 Inode 테이블의 블록 포인터가 즉시 초기화되어 디바이스 상에서 파일 포인팅 흔적을 잃었으나, 파일시스템의 백그라운드 트랜잭션을 로깅하는 **저널링 디바이스(JBD2)** 내에 해당 아카이브 파일이 생성 및 동기화 처리될 때 복사되었던 디스크 물리 블록들의 잔재가 여전히 존재할 가능성이 높습니다. 수집된 저널 영역의 디스크 덤프 파일인 `ext4_journal_tar.bin`을 분석하여 **삭제 유실되었던 TAR 아카이브 파일을 복구하고 내부의 플래그 텍스트**를 획득하십시오.

## 2. 제공 파일
* `ext4_journal_tar.bin` (피해 ext4 디바이스의 저널 링 버퍼 영역 전체를 통째로 카빙해 낸 원시 바이너리 덤프 파일)

## 3. 문제 목표
ext4 저널 디바이스(Inode 8)에 시스템 파일 쓰기 및 수정 동작 시 복사 적재되는 저널 트랜잭션 데이터 블록의 성질을 이해하고, 파일 카빙 도구(`binwalk`, `foremost`) 또는 수동 헥스 스캔을 통해 TAR 포맷 고유 시그니처 구조(헤더 블록 오프셋 257 위치의 **`ustar`** 매직 시그니처)를 검출해 정상 파일로 디스크 카빙 복구해 냅니다.

## 4. 의도한 풀이 흐름
1. **바이너리 매직 헤더 스캔**:
   * 제공된 `ext4_journal_tar.bin` 파일 내에 압축 파일 조각이 잔재하는지 확인하기 위해 파일 카빙 분석 도구인 `binwalk`를 가동해 정적 시그니처 맵을 출력합니다:
     ```bash
     binwalk ext4_journal_tar.bin
     ```
   * 분석 결과, 바이너리 중간 특정 오프셋 부근에서 TAR 아카이브 시작 시그니처 포맷인 **`ustar`** 매직 아스키 지시자가 탐지됨을 파악합니다.
2. **TAR 파일 카빙 (Carving) 수행**:
   * **도구 활용**: `foremost` 또는 `binwalk` 추출 옵션을 사용하여 저널 바이너리 내부의 임베디드 파일들을 자동 분리해 냅니다:
     ```bash
     binwalk -e ext4_journal_tar.bin
     ```
     또는
     ```bash
     foremost -t tar -i ext4_journal_tar.bin -o output_dir
     ```
   * **수동 카빙**: 헥스 에디터로 파일을 열고 `ustar` 매직이 포함된 512바이트 헤더 블록의 시작점(통상 `ustar`가 나타나는 오프셋 257 이전의 파일명 시작점)을 찾아, TAR 포맷이 완료되는 물리 종단(연속된 두 개의 512바이트 크기 0 패딩 빈 블록 영역)까지를 복제 커팅하여 `exfil.tar` 파일로 복원 저장합니다.
3. **압축 해제 및 플래그 획득**:
   * 카빙되어 정상적으로 복구된 TAR 아카이브 파일의 압축을 풉니다:
     `tar -xvf exfil.tar`
   * 압축이 정상 해제되어 추출된 `flag.txt` 텍스트 파일을 열고 내부 플래그를 확보합니다:
     `picoCTF{ext4_journ4l_tar_f1l3_c4rv3d}`

## 5. 정답(플래그) 규칙 및 예시
* **플래그 형식**: `picoCTF{<flag_string>}`
* **예시**: `picoCTF{ext4_journ4l_tar_f1l3_c4rv3d}`

## 6. 출제자 노트 & 파일 제작 가이드
* **아티팩트 제작 방법**:
  1. 기밀 플래그 파일 `flag.txt`를 생성합니다:
     `echo "picoCTF{ext4_journ4l_tar_f1l3_c4rv3d}" > flag.txt`
  2. 해당 파일을 tar 포맷으로 아카이빙 패킹합니다:
     `tar -cvf exfil.tar flag.txt`
  3. 테스트 ext4 볼륨에 해당 `exfil.tar` 파일을 복사 적재한 뒤, 디스크 동기화(`sync`)를 수행해 저널 데이터 디스크립터 영역에 TAR 이진 스트림 전체가 온전히 라이트(Write Log)되도록 유도합니다.
  4. 파일을 강제 삭제합니다: `rm exfil.tar; sync`
  5. `debugfs` 툴의 `dump <8> ext4_journal_tar.bin` 명령을 실행해 저널 영역 바이트 전체를 적출합니다.
  6. 덤프된 저널 이미지 내에 `ustar` 헤더 정보가 손상 없이 유지되어 기계적 카빙 복구가 성공하는지 최종 검수한 후 학생용 챌린지 파일로 유포합니다.
* **출제 포인트**: 
  * 리눅스 디스크 포렌식에서 안티 포렌식(파일 완전 삭제) 위협 발생 시, 단순 Inode 목록 복구 툴의 한계를 저널링 시스템 트랜잭션 블록 카빙 기법(ext4 Journal Tar Carving)으로 우회 돌파하여, 별도의 zlib/gzip 압축이 가미되지 않은 순수 아카이브 헤더의 512바이트 블록 정렬 특징을 활용해 기밀 증적을 환원하는 실무 능력을 측정합니다.

## 7. 트러블슈팅 및 힌트
* **Q. TAR 파일 카빙 시 헤더 시작 지점과 매직 시그니처의 정확한 오프셋 관계는 어떻게 되나요?**
  * A. TAR 규격은 512바이트 크기의 헤더 블록들로 레코드를 구성합니다. 개별 파일 아카이빙 시 해당 파일 메타데이터(파일명 등) 헤더 블록의 **오프셋 257 (0x101)** 위치에 5바이트 크기의 **`ustar`** (`75 73 74 61 72`) 문자열이 시그니처로 할당됩니다. 따라서 `ustar` 문자열을 발견했다면, 그 위치로부터 거꾸로 257바이트를 마이너스한 오프셋 지점이 실질적인 TAR 파일의 물리 시작점이 되므로 수동 카빙 시 오프셋 역연산을 정확히 기산해야 합니다.
* **Q. 카빙 복구한 TAR 파일을 해제할 때 'tar: Unexpected EOF in archive' 오류가 납니다.**
  * A. 저널 영역에서 파일 조각을 덤프할 때 파일의 물리적 끝부분에 도달하기 전 잘라내었거나, 중간 블록에 다른 시스템 로그 파일 트랜잭션이 끼어 들어간 경우 아카이브 구조가 깨집니다. TAR 명세상 아카이브의 끝은 **연속된 두 개의 512바이트 크기 Null(0x00으로만 가득 찬) 블록(총 1024바이트의 0)**으로 규정되어 있습니다. 이 종단 0패딩 블록이 분실되었을 경우 tar 해제 유틸리티가 경고를 내므로, 헥스 스캔 시 마지막 파일 아카이브 바디 뒤에 1024바이트의 0이 충분히 포함되도록 카빙 크기를 넉넉하게 재단해야 오류를 극복할 수 있습니다.

## 8. 학습 포인트
* **TAR 아카이브 포맷 스펙**: 512바이트 블록 구조, ustar 시그니처 위치(오프셋 257) 및 종단 Null 블록 식별 규격을 이해합니다.
* **비할당/저널 영역 디스크 카빙**: 파일시스템 카빙 유틸리티(Binwalk, Foremost)의 작동 원리를 파악하고, 저널링 트랜잭션 데이터를 역추적해 기밀 파일 아카이브를 환원하는 분석력을 구축합니다.
