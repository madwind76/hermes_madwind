# 손상된 GZIP 헤더의 정상화 (Corrupted GZIP Header Repair)

> **난이도**: 초급  
> **소요 시간**: 15~20분  
> **참고 picoCTF 문제**: 파일 시그니처 손상 복구 (GZIP Header 수동 복원)

## 1. 배경 시나리오
침해 사고 추적 중, 시스템 내 `/tmp` 폴더의 일시 보존 공간에서 백도어의 원본 소스로 추정되는 압축 아카이브 파일 `flag_archive.tar.gz`가 획득되었습니다. 하지만 공격자가 보안 유틸리티의 시그니처 기반 정적 탐지를 우회하기 위해 해당 압축 파일의 **매직 헤더 바이너리를 임의의 문자열로 손상 변조**해 놓은 것이 발견되었습니다. `tar -zxvf` 명령어 및 일반 압축 해제 뷰어로 압축 풀기를 시도하면 `not in gzip format` 오류가 발생합니다. GZIP 아카이브 규격에 맞게 파일 헤더 첫 부분을 직접 수리하여 정상 압축 해제 후 플래그를 획득하십시오.

## 2. 제공 파일
* `flag_archive.tar.gz` (첫 부분 매직 바이너리가 임의의 헥스 값으로 훼손된 GZIP 압축 파일)

## 3. 문제 목표
GZIP 파일 포맷 규격(RFC 1952)에 따른 파일 헤더 시그니처 구조(Magic Bytes `1F 8B` 및 Compression Method `08`)를 이해하고, 헥스 에디터로 오염된 첫 3바이트 값을 바르게 패치하여 원본 압축 해제를 승인받습니다.

## 4. 의도한 풀이 흐름
1. **오류 상황 진단**:
   * 터미널에서 `file flag_archive.tar.gz` 명령어를 구동합니다. 결과가 `data` 형식으로 나타나 시스템이 이를 아카이브로 인식하지 못함을 확인합니다.
   * `tar -zxvf flag_archive.tar.gz` 실행 시 `tar: child returned status 1`, `gzip: stdin: not in gzip format` 등의 에러 메시지가 검출됩니다.
2. **훼손 헤더 헥스 분석**:
   * 헥스 에디터로 `flag_archive.tar.gz`를 엽니다.
   * 시작 바이트 영역을 확인합니다:
     현재: `41 42 43 00 00 ...` (아스키 `ABC` 데이터가 선언되어 있음)
3. **GZIP 시그니처 패치**:
   * GZIP 파일의 표준 규격 명세는 다음과 같습니다:
     * 첫 바이트 (오프셋 0): `1F` (GZIP ID1)
     * 두 번째 바이트 (오프셋 1): `8B` (GZIP ID2)
     * 세 번째 바이트 (오프셋 2): `08` (압축 메소드 - Deflate)
   * 헥스 에디터 상에서 선두의 잘못된 3바이트(`41 42 43`)를 표준 매직 바이트인 `1F 8B 08`로 변환 패치합니다.
4. **압축 정상화 및 추출**:
   * 패치한 파일을 `repaired.tar.gz`로 저장합니다.
   * 정상적으로 인식되는지 `file repaired.tar.gz`로 확인합니다. `gzip compressed data` 문구가 출력되는 것을 검증합니다.
   * 압축을 해제합니다:
     `tar -zxvf repaired.tar.gz`
   * 압축이 정상 해제되어 튀어나온 `flag.txt` 파일을 열어 최종 플래그를 획득합니다.
     (최종 플래그: `picoCTF{gzip_m4g1c_bytes_r3st0r3d}`)

## 5. 정답(플래그) 규칙 및 예시
* **플래그 형식**: `picoCTF{<flag_string>}`
* **예시**: `picoCTF{gzip_m4g1c_bytes_r3st0r3d}`

## 6. 출제자 노트 & 파일 제작 가이드
* **아티팩트 제작 방법**:
  1. 기밀 텍스트 파일 `flag.txt`에 `picoCTF{gzip_m4g1c_bytes_r3st0r3d}`를 작성합니다.
  2. 리눅스 표준 압축 명령을 사용해 아카이브를 만듭니다:
     `tar -zcvf flag_archive.tar.gz flag.txt`
  3. 헥스 에디터를 실행하여 생성된 파일의 첫 3바이트 값인 `1F 8B 08`을 아스키 문자열 `ABC`에 해당하는 `41 42 43`으로 강제 덮어씁니다.
  4. 저장한 파일을 배포 파일로 지정합니다.
* **출제 포인트**: 
  * 바이너리 분석에서 가장 기본이자 강력한 필터링 요소인 파일 서명(File Signature)의 복원 원리를 터득하고, 파일 확장자에 의존하지 않고 실제 매직 바이트 기반으로 아티팩트의 본래 포맷을 감지하는 습관을 기릅니다.

## 7. 트러블슈팅 및 힌트
* **Q. 헤더를 고쳤는데도 데이터 압축 에러가 납니다.**
  * A. 오프셋 3 이후의 헤더 필드(플래그 바이트, 수정 시각 등)나 파일 테일의 CRC-32 체크값, 원본 비압축 크기(ISIZE) 오프셋 영역이 훼손되었을 가능성이 있습니다. 본 초급 챌린지는 첫 3바이트만 조작되었으므로, 오프셋 0~2의 바이트 수정에만 집중해 보십시오.
* **Q. 윈도우 환경에서 파일 첫 부분을 고치기 쉬운 도구가 있나요?**
  * A. HxD, 010 Editor 등의 무료 헥스 뷰어 프로그램을 가동하여 직접 바이트 입력창에 `1F 8B 08`을 덮어써서 간편하게 수정할 수 있습니다.

## 8. 학습 포인트
* **파일 시그니처(File Signature) 이론**: 운영체제와 카빙 도구가 특정 파일의 유형을 판별하기 위해 참조하는 선두 매직 바이트 데이터 구성을 규명합니다.
* **GZIP 파일 규격**: 아카이브 헤더 내부 구조(RFC 1952 사양) 및 파일 무결성 복구 기법을 실증 실습합니다.
