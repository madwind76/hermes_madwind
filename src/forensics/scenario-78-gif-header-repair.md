# 변조된 GIF 헤더 및 컬러 테이블 복원 (Corrupted GIF Header & Logical Screen Descriptor)

> **난이도**: 초중급  
> **소요 시간**: 25~30분  
> **참고 picoCTF 문제**: 손상된 미디어 헤더 복원 및 디코드 (GIF89a 명세 분석)

## 1. 배경 시나리오
데이터 유출 용의자로부터 확보한 이미지 증적 파일인 `corrupted.gif`를 이미지 리더나 브라우저를 통해 실행하면 "지원하지 않는 형식이거나 가로/세로 해상도가 유효하지 않음" 에러가 뜨며 파일 렌더링에 실패합니다. 헥스 에디터로 정밀 분석한 결과, 피의자는 보안 솔루션의 파일 카빙 탐지를 기만하고 이미지를 파쇄하기 위해 파일의 시작 **매직 시그니처(Signature)** 6바이트와 논리 화면 크기를 명시하는 **논리 화면 디스크립터(Logical Screen Descriptor)** 영역의 가로/세로 해상도 값을 고의로 소거(`00 00`)해 둔 상태입니다. 원본 이미지의 픽셀 해상도가 가로 **400px**, 세로 **300px**이라는 제보 정보를 바탕으로 파일 바이너리를 수작업으로 정상 수리하여 **이미지 복원 후 노출되는 플래그**를 획득하십시오.

## 2. 제공 파일
* `corrupted.gif` (선두 시그니처 및 논리 화면 디스크립터 해상도 속성 영역이 0으로 오염되어 깨진 GIF 파일)

## 3. 문제 목표
GIF 파일 포맷 명세(선두 6바이트 매직 시그니처 `GIF87a`/`GIF89a`, 논리 화면 디스크립터 영역의 2바이트 리틀엔디언 가로/세로 해상도 변수 저장 방식)를 이해하고, 헥스 에디터를 이용해 고의 오염된 오프셋 영역을 아키텍처에 맞게 바이트 패칭(Hex Patching)합니다.

## 4. 의도한 풀이 흐름
1. **바이너리 정밀 분석 및 오류 판별**:
   * 제공된 `corrupted.gif` 파일을 헥스 에디터로 로드합니다.
   * 오프셋 0~5 (최초 6바이트): 현재 `00 00 00 00 00 00` (또는 `42 41 44 47 49 46` 등 깨진 바이트)로 채워져 GIF 시그니처가 상실된 상태입니다.
   * 오프셋 6~9 (논리 화면 디스크립터 크기): 가로(2바이트), 세로(2바이트)가 현재 `00 00` 및 `00 00`으로 훼손되어 리더가 렌더링 영역을 확보할 수 없어 에러가 남을 진증합니다.
2. **시그니처 및 해상도 헥스 패치 복원**:
   * **시그니처 복구**: 오프셋 0~5 영역을 GIF 표준 규격인 **`47 49 46 38 39 61`** (아스키 문자 **`GIF89a`**)로 덮어씁니다.
   * **가로 해상도(400px) 복구**:
     * 400을 16진수로 변환하면 `01 90` 입니다.
     * GIF 사양은 정수 필드를 **리틀 엔디언(Little Endian)**으로 저장하므로, 오프셋 6~7 영역에 바이트 스왑된 **`90 01`** 을 기입합니다.
   * **세로 해상도(300px) 복구**:
     * 300을 16진수로 변환하면 `01 2C` 입니다.
     * 마찬가지로 리틀 엔디언 형식을 적용하여 오프셋 8~9 영역에 바이트 스왑된 **`2C 01`** 을 기입합니다.
   * 바이트 수리를 완료하고 `repaired_flag.gif` 파일로 내보내기합니다.
3. **이미지 렌더링 및 플래그 획득**:
   * 복구된 `repaired_flag.gif` 파일을 이미지 뷰어로 켭니다.
   * 에러가 완벽히 수리되어 정상 이미지 화면이 출력되고 내부에 박제 기입된 최종 플래그를 식별합니다:
     `picoCTF{gif_89a_scr33n_d3scr1pt0r_fixed}`

## 5. 정답(플래그) 규칙 및 예시
* **플래그 형식**: `picoCTF{<flag_string>}`
* **예시**: `picoCTF{gif_89a_scr33n_d3scr1pt0r_fixed}`

## 6. 출제자 노트 & 파일 제작 가이드
* **아티팩트 제작 방법**:
  1. 기밀 플래그 `picoCTF{gif_89a_scr33n_d3scr1pt0r_fixed}` 문자열을 수록한 가로 400px, 세로 300px 해상도의 단색 이미지를 제작하고 GIF 형식으로 저장합니다.
  2. 헥스 에디터로 GIF 파일을 엽니다.
  3. 오프셋 0~5 위치의 매직 바이트 `47 49 46 38 39 61` (GIF89a)을 전부 `00`으로 덮어씁니다.
  4. 이어서 오프셋 6~7 위치의 가로 해상도 데이터 `90 01` 및 오프셋 8~9 위치의 세로 해상도 데이터 `2C 01` 영역도 전부 `00 00`으로 덮어써서 구조를 변조합니다.
  5. 손상된 파일을 `corrupted.gif` 이름으로 포장하여 학생 배포 아티팩트로 제공합니다.
* **출제 포인트**: 
  * 파일 식별 정보뿐 아니라 파일 포맷이 요구하는 세부 속성 데이터 블록(GIF Logical Screen Descriptor Specifications) 명세에 대한 리틀엔디언 정수 보존 규칙을 이해하고, 정밀 바이트 교정(Little Endian Hex Patching)으로 파일 렌더링 엔진의 작동 규격을 원천 복원하는 능력을 훈련시킵니다.

## 7. 트러블슈팅 및 힌트
* **Q. 리틀 엔디언 변환을 무시하고 빅 엔디언(01 90 / 01 2C)으로 기입하면 어떻게 되나요?**
  * A. GIF 디코딩 라이브러리는 해당 바이트를 리틀엔디언으로 자동 해석합니다. 즉, `01 90`으로 기입하면 가로 해상도를 `0x9001` (36,865 픽셀)로, `01 2C`로 기입하면 세로 해상도를 `0x2C01` (11,265 픽셀)로 해석하여 실제 이미지 크기보다 지나치게 비대한 영역 생성을 시도하게 됩니다. 이 경우 메모리 할당 에러(OOM)가 나거나 깨진 선들만 렌더링되므로 반드시 아키텍처 바이트 순서(Byte Order)를 교차 검수해야 합니다.
* **Q. 파일 내부의 컬러 테이블(Color Table) 정보도 손상되었는지 알려면 어떻게 하나요?**
  * A. GIF 파일은 논리 화면 디스크립터 바로 다음 영역인 오프셋 10 위치에 1바이트 크기의 **Packed Fields** 속성을 지닙니다. 이 바이트의 최상위 비트(Global Color Table Flag)가 `1`로 세팅되어 있으면 전역 컬러 테이블이 기입되어 있음을 의미합니다. 본 문제는 매직 시그니처와 스크린 정보 4바이트 수리로 디코딩되게 설계되어 컬러 테이블 패킹 정보는 정상 보존되어 있습니다.

## 8. 학습 포인트
* **GIF 파일 구조 명세**: GIF87a/GIF89a 헤더 배열 구성 및 논리 화면 기술 블록(Logical Screen Descriptor) 구조에 대해 상세 이해합니다.
* **리틀 엔디언 바이트 스왑**: 이진 매개변수 변조 대응 시, 대상 파일 사양이 요구하는 바이트 정렬 순서(Endianness)를 연산하고 헥스 패치 교정을 실행하는 분석가 능력을 갖춥니다.
