# 리눅스 배시 명령어 스톰핑 분석 (Bash Command Line Obfuscation)

> **난이도**: 초중급  
> **소요 시간**: 20~25분  
> **참고 picoCTF 문제**: 원시 터미널 입출력 기록 분석 및 해독 (Bash Backspace 제어 문자 우회)

## 1. 배경 시나리오
보안 모니터링 로그 관제 시스템을 우회하기 위해, 공격자는 터미널 셸 세션 상에서 명령어를 타이핑할 때 제어 문자(Control Characters)를 교묘하게 활용하는 **명령행 스톰핑(Command Line Stomping)** 기법을 활용했습니다. 공격자는 터미널 세션 로깅 유틸리티가 작동 중임을 인지하고, 민감한 플래그 출력 명령어를 입력한 뒤 즉시 백스페이스(Backspace) 제어 바이트(`\x08` 또는 `\x7f`)를 다수 인가하여 화면상에는 평범한 가짜 명령어(예: `echo "hello world"`)만 노출되도록 위장했습니다. 분석가가 격리 수집한 원시 터미널 레코딩 로그 파일 `session.log`를 분석하여 **용의자가 화면에서 지워 숨겼던 실제 실행된 플래그 명령어**를 복원하십시오.

## 2. 제공 파일
* `session.log` (리눅스 `script` 명령어로 생성된 터미널 세션 원시 바이너리 로그 파일)

## 3. 문제 목표
리눅스 및 유닉스 의사 터미널(TTY)의 제어 기호 규칙(특히 백스페이스 `\x08` / `\x7f` 및 캐리지 리턴 `\r`)이 터미널 에뮬레이터 화면 상에 렌더링될 때의 시각적 왜곡 동작을 이해하고, 원시 바이너리 덤프 파싱을 통해 원본 키 입력 문자열 시퀀스를 역추적합니다.

## 4. 의도한 풀이 흐름
1. **시각적 왜곡 상태 관찰**:
   * 터미널에서 `cat session.log` 명령을 실행해 화면 출력을 확인합니다.
   * 화면에는 단순히 `$ echo "hello world"` 라는 평범한 명령의 출력 결과만 보입니다.
2. **원시 바이트 분석 (Hex 덤프)**:
   * 헥스 에디터로 `session.log` 파일을 열거나 `xxd` 명령으로 바이너리를 덤프합니다:
     ```bash
     xxd session.log
     ```
   * 덤프 결과, 평범한 텍스트 사이에 백스페이스 제어 바이트인 `08` (또는 `7f`)이 대량으로 섞여 있는 비정상 구조를 식별합니다.
3. **타이핑 시퀀스 역추적**:
   * 원시 헥스 바이트 스트림의 타이핑 흐름을 역추적합니다:
     1. `echo "picoCTF{b4sh_b4cksp4c3_cl34nup_byp4ss}"` 문자열을 먼저 타이핑함.
     2. 이어서 `08` (Backspace) 제어 바이트가 42회 반복 인가됨 (화면상에서 해당 명령 문자열이 뒤에서부터 한 글자씩 모두 지워짐).
     3. 이어서 `echo "hello world"` 문자열을 새로 타이핑함.
     4. 엔터(`0d` 또는 `0a`)를 입력해 실행함.
   * *주: 셸 엔진은 입력 버퍼를 해석할 때 백스페이스 기호를 만나면 버퍼의 마지막 글자를 소거하므로, 결과적으로 실제로 실행된 명령어는 버퍼에 온전히 살아 남아 실행되었던 `echo "picoCTF{b4sh_b4cksp4c3_cl34nup_byp4ss}"` 였습니다.*
4. **플래그 획득**:
   * 백스페이스로 화면상에서 지워져 감춰졌던 원본 플래그 값을 획득합니다:
     `picoCTF{b4sh_b4cksp4c3_cl34nup_byp4ss}`

## 5. 정답(플래그) 규칙 및 예시
* **플래그 형식**: `picoCTF{<flag_string>}`
* **예시**: `picoCTF{b4sh_b4cksp4c3_cl34nup_byp4ss}`

## 6. 출제자 노트 & 파일 제작 가이드
* **아티팩트 제작 방법**:
  1. 리눅스 가상 머신에서 터미널 레코딩을 가동합니다:
     `script session.log`
  2. 셸 세션 프롬프트가 뜨면 먼저 실제 플래그 명령을 입력하되 엔터는 누르지 않습니다:
     `echo "picoCTF{b4sh_b4cksp4c3_cl34nup_byp4ss}"`
  3. 이어서 백스페이스 키를 꾹 눌러 프롬프트 라인의 글자들을 전부 지웁니다. (이때 터미널 입출력 스트림에는 `08` 또는 `7f` 바이트가 순차 누적됩니다)
  4. 다 지워진 상태에서 가짜 명령어를 치고 엔터를 눌러 실행합니다:
     `echo "hello world"`
  5. `exit`를 쳐서 레코딩을 종료하고 저장된 `session.log`를 아티팩트로 배포합니다.
* **출제 포인트**: 
  * 정적 텍스트 감사 로그와 실시간 시스템 입력 스트림의 렌더링 메커니즘 차이를 이해하고, 제어 문자(Control Codes)를 악용한 시각적 기만 행위(Stomping/Obfuscation)에 대해 원시 헥스 바이트 스트림 분석으로 완벽히 대응하는 기법을 습득합니다.

## 7. 트러블슈팅 및 힌트
* **Q. nano나 vi 에디터로 session.log를 열었을 때는 제어 문자가 어떻게 보이나요?**
  * A. 텍스트 에디터 설정에 따라 다르지만, 일반적으로 백스페이스 기호 `\x08`은 `^H` 기호로 표시됩니다. 따라서 파일 중간에 `^H^H^H^H...` 와 같이 반복되는 문자열이 관찰된다면 그 앞의 문자들이 백스페이스로 화면상에서 소거되었음을 정적으로 즉시 판단할 수 있습니다.
* **Q. 이 제어 기호를 자동으로 정제하여 화면 출력 상태가 아닌 실제 타이핑 흐름대로 시뮬레이션해 주는 분석 스크립트는 어떻게 짜나요?**
  * A. 파이썬에서 파일 바이트를 읽어들이면서, `0x08` 또는 `0x7F` 바이트를 만나면 리스트 버퍼의 마지막 요소를 `pop()` 시키고, 일반 아스키 바이트를 만나면 `append()` 시켜 최종적으로 축적된 버퍼만 조인(Join)해 출력하게 코딩하면 실제 셸 인터프리터가 실행한 명령어 원형을 한눈에 안전하게 확인할 수 있습니다.

## 8. 학습 포인트
* **터미널 제어 기호 규칙 (ANSI Control Characters)**: CLI 환경에서 화면 제어(커서 이동, 문자 소거 등)에 활용되는 저수준 제어 문자 명세를 이해합니다.
* **텍스트 스톰핑 우회**: 텍스트 뷰어(cat, less 등)의 렌더링 맹점을 노린 안티 포렌식 기법을 정밀 헥스 바이트 시퀀스 대조 분석으로 극복하는 능력을 갖춥니다.
