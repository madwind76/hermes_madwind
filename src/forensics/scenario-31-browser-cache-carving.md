# 브라우저 캐시 이미지 복구 (Browser Cache Carving)

> **난이도**: 초중급  
> **소요 시간**: 20~25분  
> **참고 picoCTF 문제**: 브라우저 흔적 추적 및 미디어 카빙 (Chrome Cache 파일 복구 실무)

## 1. 배경 시나리오
사내 기밀 제품의 설계도 도면이 웹 브라우저를 통해 무단 조회되었다는 보안 경보가 있었습니다. 용의자는 해당 사이트에 접속한 직후 웹 브라우저 방문 기록(History)을 모두 지우고 발뺌하고 있으나, 크롬(Chrome) 브라우저의 임시 보존 스토리지인 캐시(Cache) 영역까지는 소거하지 못한 것으로 드러났습니다. 조사팀은 크롬 캐시 디렉터리(`~/.config/google-chrome/Default/Cache/Cache_Data/`)에서 원시 데이터 블록 파일들을 수집했습니다. 이 파일들을 스캔 및 카빙하여 당시에 렌더링되어 로드되었던 플래그 이미지를 복구해야 합니다.

## 2. 제공 파일
* `chrome_cache.zip` (크롬 브라우저 캐시 원시 블록 파일들이 압축된 파일)
  * 주요 포함 파일: `data_0`, `data_1`, `data_2`, `f_00001a`, `f_0000c5` 등

## 3. 문제 목표
크롬 브라우저의 캐시 저장 메커니즘(Blockfile Cache 또는 Simple Cache의 파일 구조)을 이해하고, 캐시 데이터 파일 내부에 섞여 있는 HTTP 헤더 메타데이터를 분리하여 원본 이미지 파일(PNG/JPEG 시그니처)을 카빙 및 복원해 냅니다.

## 4. 의도한 풀이 흐름
1. **캐시 파일 아카이브 해제**:
   * 제공된 `chrome_cache.zip` 압축을 해제하고 추출된 파일 목록을 검사합니다.
   * `f_`로 시작하는 개별 캐시 파일들과 대용량 블록 파일(`data_0` 등)을 확인합니다.
2. **시그니처 기반 이미지 추출 (File Carving)**:
   * 캐시된 파일들은 웹에서 내려받은 원본 파일의 앞부분에 HTTP 응답 헤더(HTTP/1.1 200 OK 등)가 덧붙여진 상태로 저장됩니다.
   * `binwalk` 도구를 사용해 캐시 데이터 내에 PNG 또는 JPEG 파일 시그니처가 내장되어 있는지 일괄 검색합니다:
     ```bash
     binwalk chrome_cache/*
     ```
   * 검색 결과 특정 캐시 엔트리 파일(예: `chrome_cache/f_0000c5`)의 오프셋 주소에서 PNG 이미지 시그니처(`89 50 4E 47`)가 시작되고 있음을 식별합니다.
3. **바이너리 카빙 수행**:
   * `binwalk -e chrome_cache/f_0000c5` 명령어 또는 `foremost`를 사용하여 해당 오프셋 뒤편의 PNG 파일을 물리 추출합니다.
   * 수동 풀이 시, 헥스 에디터로 `f_0000c5` 파일을 열고 처음 마주하는 PNG 매직 바이트 `89 50 4E 47` 이전의 HTTP 헤더 바이트들을 삭제한 뒤 `recovered.png`로 저장합니다.
4. **결과 확인**:
   * 복구 완료된 `recovered.png` 이미지를 실행하여 본문 중앙에 기록되어 있는 최종 플래그를 획득합니다.
     (최종 플래그: `picoCTF{chrome_cache_image_recovered}`)

## 5. 정답(플래그) 규칙 및 예시
* **플래그 형식**: `picoCTF{<flag_string>}`
* **예시**: `picoCTF{chrome_cache_image_recovered}`

## 6. 출제자 노트 & 파일 제작 가이드
* **아티팩트 제작 방법**:
  1. 기밀 플래그 `picoCTF{chrome_cache_image_recovered}` 텍스트가 명시된 PNG 도면 이미지 `design.png`를 작성합니다.
  2. 로컬 웹 서버를 구동하고 해당 이미지를 호스트합니다.
  3. 크롬 브라우저를 띄워 해당 이미지 링크로 직접 접속하여 웹 서핑을 진행합니다.
  4. 브라우저를 종료하고 크롬 캐시 디렉터리로 이동하여 당일 날짜에 수정된 `f_xxxxxx` 형식의 캐시 파일들을 검색합니다.
  5. `file` 명령어로 뒤져 헤더가 HTTP 응답이고 내부에 PNG 시그니처가 매핑된 파일을 탐색 및 정제하여 아티팩트 `f_0000c5`로 등록 및 배포합니다.
* **출제 포인트**: 
  * 사용자가 방문한 도메인뿐만 아니라, 브라우저가 오프라인 환경 작동을 위해 임시 디스크에 저장하는 실질적 렌더링 파일 콘텐츠(Cache Data Carving)를 발굴해 내는 정밀 브라우저 조사 역량을 강화합니다.

## 7. 트러블슈팅 및 힌트
* **Q. 캐시 파일 확장자를 그냥 .png로 바꿨더니 그림 파일 포맷 에러가 나면서 열리지 않습니다.**
  * A. 크롬 캐시 파일은 실제 PNG 바이트 바로 앞에 서버가 전송한 HTTP 헤더(`HTTP/1.1 200 OK\r\nContent-Type: image/png\r\n...`) 정보가 먼저 입력되어 있습니다. 이 헤더가 지워지지 않고 남아 있으면 이미지 디코더가 파일 시그니처 매칭에 실패하므로, 반드시 헥스 에디터나 카빙 툴로 앞의 헤더 텍스트를 제거하고 저장해야 정상 작동합니다.
* **Q. 크롬 캐시를 전문적으로 디코딩해 주는 도구는 없나요?**
  * A. NirSoft에서 배포하는 **ChromeCacheView** 유틸리티를 사용하면 캐시 원본 파일의 경로, 다운로드 URL, 서버 응답 시간 정보와 실제 캐시 바이너리를 GUI 형태로 손쉽게 일괄 분석 및 카빙해 낼 수 있어 실무 분석 시 매우 유용합니다.

## 8. 학습 포인트
* **브라우저 캐시 아키텍처**: 크롬/파이어폭스 등의 브라우저가 성능 극대화를 위해 미디어 및 정적 리소스를 HTTP 메타데이터와 결합하여 로컬 스토리지에 유지하는 원리를 학습합니다.
* **HTTP 헤더 가비지 처리**: 파일 포맷 구조상 결합된 이종 데이터 스트림에서 헤더 시그니처(Magic Bytes) 오프셋을 역산해 낸 뒤 불필요한 네트워크 제어 정보를 잘라내고 데이터만 수렴하는 카빙 원리를 습득합니다.
