# 조작된 압축 파일의 크기 오류 (ZIP Archive Size Anomaly)

> **난이도**: 초중급  
> **소요 시간**: 20~25분  
> **참고 picoCTF 문제**: ZIP 아카이브 수동 카빙 및 무결성 복원 (ZIP Size metadata 조작 극복)

## 1. 배경 시나리오
보안 솔루션의 파일 크기 감사 필터링 규정을 우회하기 위해, 공격자가 유출 파일인 `anomaly.zip` 내부에 중요 텍스트를 압축 저장하면서 헤더 데이터 중 **압축 크기(Compressed Size)**와 **비압축 크기(Uncompressed Size)** 정보를 인위적으로 아주 작게 조작했습니다. 압축 프로그램을 실행하여 해제를 시도하면 일부 텍스트만 잘려 출력되거나 CRC 체크 에러가 발생하여 온전한 파일을 획득할 수 없습니다. 이 ZIP 파일의 헤더를 직접 수동으로 정렬 및 수리하여 전체 원본 플래그 문서를 정상 복구해야 합니다.

## 2. 제공 파일
* `anomaly.zip` (압축 크기 정보 메타데이터가 고의로 축소 변조된 ZIP 파일)

## 3. 문제 목표
ZIP 아카이브의 바이너리 포맷(특히 Local File Header 및 Central Directory Entry 구조체) 중 파일 크기 파라미터 오프셋을 파악하고, 실제 바이너리 데이터를 실측하여 모순된 크기 값을 헥스 패치(Hex Patch)함으로써 손상된 아카이브를 정상 렌더링하도록 수리합니다.

## 4. 의도한 풀이 흐름
1. **문제점 식별**:
   * `unzip anomaly.zip` 명령어를 실행하면 `warning [anomaly.zip]:  expected 38 bytes, got 5` 와 같은 경고가 발생하거나, 압축을 해제해 나온 `flag.txt` 파일을 열었을 때 `picoC` 와 같이 앞의 5글자만 잘려서 출력됨을 확인합니다.
2. **로컬 파일 헤더 분석 (Local File Header)**:
   * 헥스 에디터로 `anomaly.zip`을 엽니다.
   * 첫 4바이트 시그니처 `50 4B 03 04` (PK\x03\x04)를 확인합니다.
   * 로컬 헤더 내에서 다음 오프셋을 점검합니다:
     * 오프셋 18: Compressed Size (4바이트) -> 현재 `05 00 00 00` (5바이트)
     * 오프셋 22: Uncompressed Size (4바이트) -> 현재 `05 00 00 00` (5바이트)
   * 파일명 길이(오프셋 26, `08 00` -> 8바이트인 `flag.txt`)와 추가 필드 길이를 지나 실제 데이터가 시작되는 위치를 확인합니다.
   * 헥스 덤프 상에 평문 상태로 실재하는 원본 데이터 문자열(`picoCTF{zip_h3ad3r_s1z3_m1sm4tch_byp4ss}`)의 길이를 실측합니다. 실측 결과 원본 크기는 총 38바이트(16진수 `0x26`)임을 판정합니다.
3. **바이너리 값 수리**:
   * **로컬 파일 헤더 수정**:
     * 오프셋 18의 값을 `26 00 00 00`으로 변경합니다.
     * 오프셋 22의 값을 `26 00 00 00`으로 변경합니다.
   * **중앙 디렉터리 헤더 (Central Directory Header) 수정**:
     * 파일 하부의 중앙 디렉터리 시그니처 `50 4B 01 02` (PK\x01\x02) 영역을 식별합니다.
     * 중앙 디렉터리 오프셋 20 (Compressed Size)과 오프셋 24 (Uncompressed Size)를 찾아 동일하게 `26 00 00 00`으로 패치합니다.
4. **결과 확인**:
   * 패치한 내용을 `repaired.zip`으로 저장하고 압축을 풉니다:
     `unzip repaired.zip`
   * 잘리는 영역 없이 온전하게 추출된 `flag.txt` 파일을 열어 최종 플래그를 확인합니다.
     (최종 플래그: `picoCTF{zip_h3ad3r_s1z3_m1sm4tch_byp4ss}`)

## 5. 정답(플래그) 규칙 및 예시
* **플래그 형식**: `picoCTF{<flag_string>}`
* **예시**: `picoCTF{zip_h3ad3r_s1z3_m1sm4tch_byp4ss}`

## 6. 출제자 노트 & 파일 제작 가이드
* **아티팩트 제작 방법**:
  1. 기밀 파일 `flag.txt`에 `picoCTF{zip_h3ad3r_s1z3_m1sm4tch_byp4ss}` (총 38자)를 기록하고 압축하지 않은 무압축(Store) 방식으로 압축 파일을 생성합니다:
     `zip -0 anomaly.zip flag.txt` (무압축 방식으로 묶어야 바이트 크기 실측이 직관적입니다)
  2. 헥스 에디터로 생성된 파일을 열어 다음 위치를 강제 변조합니다:
     * 로컬 헤더(`50 4B 03 04`) 기준 오프셋 18, 22 위치의 `26 00 00 00` -> `05 00 00 00`
     * 중앙 디렉터리(`50 4B 01 02`) 기준 오프셋 20, 24 위치의 `26 00 00 00` -> `05 00 00 00`
  3. 변조된 `anomaly.zip` 파일을 저장하여 배포용 파일로 확정합니다.
* **출제 포인트**: 
  * 파일 포맷의 길이 선언자 오작동(Integer/Metadata Manipulation)을 활용한 시각적/기능적 정보 은닉 원리를 이해하고, 이진 바이트 크기 실측을 통한 포렌식 바이너리 교정 역량을 함양합니다.

## 7. 트러블슈팅 및 힌트
* **Q. 압축 크기와 비압축 크기를 다르게 입력하면 어떻게 되나요?**
  * A. 압축 방식(Compression Method, 오프셋 8)이 `00 00` (무압축 Store)인 경우, 두 크기는 완벽히 일치해야 합니다. 디플레이트(Deflate, `08 00`) 방식으로 실제 압축을 수행한 이미지라면 압축 완료된 바이너리의 크기(Compressed)와 복원되었을 때의 본문 크기(Uncompressed)가 상이하므로 실측 시 압축 알고리즘의 블록 경계를 추가 검토해야 하므로 본 초급 문제에서는 무압축 상태로 정제했습니다.
* **Q. 파일명 길이는 어떻게 판별하나요?**
  * A. 오프셋 26 위치의 2바이트 값이 파일명 길이를 정의합니다. 이 크기가 정상적이어야만 파일명 이후에 시작되는 실제 페이로드 데이터의 오프셋 시작점을 올바르게 짚을 수 있습니다.

## 8. 학습 포인트
* **ZIP 파일 포맷 구조**: 로컬 헤더, 파일 데이터 영역, 중앙 디렉터리 인덱스로 이어지는 ZIP 파일 내부 데이터 매핑 체계를 습득합니다.
* **메타데이터 위조 복구**: 압축 유틸리티 파서의 크기 참조 로직 취약성을 악용해 데이터 일부를 숨기는 공격적 안티 포렌식 행위에 대해 바이트 수동 교정으로 완벽 복원하는 역량을 배양합니다.
