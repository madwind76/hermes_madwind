# 디스크 슬랙 공간의 속삭임 (Whispering Slack Space)

> **난이도**: 초중급  
> **소요 시간**: 25~30분  
> **참고 picoCTF 문제**: 저수준 파일시스템 디스크 이미지 분석 (File Slack Area 데이터 카빙)

## 1. 배경 시나리오
디지털 포렌식 팀은 기밀 유출 혐의가 있는 피의자의 USB에서 FAT32 파일시스템 디스크 이미지 `usb_disk.dd`를 수집했습니다. 파일 목록을 훑어보면 일반 텍스트 파일 `readme.txt`가 존재하고, 이 파일의 논리 크기(Logical Size)는 겨우 100바이트로 기록되어 있어 일반 편집기(notepad, cat 등)로 열어보면 짧은 환영 인사 문구만 출력됩니다. 그러나 파일시스템이 할당하는 최소 물리 블록 크기(Cluster/Block)와 실제 파일 데이터 크기 사이의 유휴 영역인 **파일 슬랙(File Slack)** 공간에 공격자가 은밀하게 기밀 플래그를 밀어 넣은 정황이 포착되었습니다.

## 2. 제공 파일
* `usb_disk.dd` (FAT32 포맷의 소형 디스크 이미지 파일, 약 10MB)

## 3. 문제 목표
파일의 논리적 크기(Logical Size)와 물리적 크기(Physical Size)의 정의 및 유휴 파일 슬랙(Slack Space)의 개념을 이해하고, 디스크 이미지 상에서 특정 파일의 데이터 블록 시작 주소를 찾아내어 해당 주소의 물리 클러스터 전체(예: 4096바이트)를 raw 헥스 덤프하여 슬랙 영역에 은닉된 플래그를 식별합니다.

## 4. 의도한 풀이 흐름
1. **타깃 파일 및 메타데이터 추적**:
   * 슬루스킷(Sleuthkit)의 `fls usb_disk.dd` 명령을 통해 `readme.txt` 파일의 아이노드(Inode) 혹은 디렉터리 엔트리 번호를 확인합니다.
     (예: `readme.txt` -> Inode 4)
2. **물리 블록 할당 정보 확인**:
   * `istat usb_disk.dd 4` 명령을 실행하여 해당 파일이 위치한 물리 클러스터(Cluster) 또는 섹터(Sector) 번호 및 오프셋을 알아냅니다.
   * 출력물 예시:
     * `Size: 100 bytes` (논리 크기)
     * `Sectors: 2048` (실제 할당된 섹터 시작 주소)
3. **바이너리 물리 오프셋 계산**:
   * 한 섹터는 512바이트이며, 파일은 섹터 2048번부터 쓰여 있습니다.
   * 물리적 바이트 시작 지점: `2048 * 512 = 1,048,576` 바이트
   * 논리 크기가 100바이트이므로, 오프셋 `1,048,576 + 100` 지점부터 해당 클러스터가 끝나는 `1,048,576 + 4096` 바이트 사이 공간은 파일 슬랙(File Slack) 영역으로 채워져 있습니다. 일반 OS는 이 영역의 데이터를 사용자에게 읽어주지 않습니다.
4. **슬랙 공간 바이너리 덤프**:
   * `dd` 명령어를 이용해 해당 물리 위치를 직접 추출합니다:
     `dd if=usb_disk.dd bs=512 skip=2048 count=8 | xxd` (1클러스터 = 8섹터 = 4096바이트)
   * 출력 결과를 점검하여, 0~100바이트 범위의 정상 `readme.txt` 본문 내용 뒤편에 존재하는 패딩 영역(주로 0x00 또는 쓰레기 값) 끝부분에 숨어 있는 `picoCTF{slack_sp4ce_h1des_secrets}` 문자열을 발견합니다.

## 5. 정답(플래그) 규칙 및 예시
* **플래그 형식**: `picoCTF{<flag_string>}`
* **예시**: `picoCTF{slack_sp4ce_h1des_secrets}`

## 6. 출제자 노트 & 파일 제작 가이드
* **아티팩트 제작 방법**:
  1. 10MB 크기의 가상 FAT32 디스크 이미지를 생성합니다:
     `dd if=/dev/zero of=usb_disk.dd bs=1M count=10`
     `mkfs.vfat usb_disk.dd`
  2. 디스크 이미지를 마운트하고 100바이트 미만의 텍스트 내용으로 `readme.txt`를 생성한 후 마운트를 해제합니다.
  3. `istat` 등을 활용해 `readme.txt`가 점유한 클러스터 주소(예: 섹터 2048)를 탐지합니다.
  4. 헥스 에디터나 파이썬 스크립트를 사용하여 디스크 이미지 파일(`usb_disk.dd`)의 오프셋 `(2048 * 512) + 100` 바이트 위치에 플래그 데이터(예: `picoCTF{slack_sp4ce_h1des_secrets}`)를 직접 덮어씁니다:
     ```python
     with open("usb_disk.dd", "r+b") as f:
         f.seek(2048 * 512 + 100)
         f.write(b"picoCTF{slack_sp4ce_h1des_secrets}")
     ```
  5. 최종 디스크 이미지를 저장하여 출제용 아티팩트로 사용합니다.
* **출제 포인트**: 
  * 파일시스템이 보장하는 논리 영역 외에, 하드디스크/플래시 미디어 물리 영역에 직접 데이터를 기록해 은닉하는 안티 포렌식 원리를 분석하는 저수준 파일시스템 디코드 역량을 평가합니다.

## 7. 트러블슈팅 및 힌트
* **Q. 디스크 이미지 전체에 대해 strings 명령을 돌리면 플래그가 한 번에 추출되지 않나요?**
  * A. 문자열 검색으로 한 번에 탐지될 수도 있습니다. 그러나 실제 CTF 환경에서는 출제자가 슬랙 영역에 들어갈 플래그 자체를 암호화하거나 인코딩(Base64, XOR 등)하는 가공 과정을 거칠 수 있으므로, 디스크의 정확한 물리 클러스터 오프셋 위치를 역산해 내는 능력이 중요시됩니다.
* **Q. 파일 슬랙과 볼륨 슬랙의 차이는 무엇인가요?**
  * A. 파일 슬랙(File Slack)은 파일 크기에 따른 할당 유휴 영역을 가리키며, 볼륨 슬랙(Volume Slack)은 전체 볼륨 파일시스템 포맷 영역 크기와 실제 파티션 물리 바이트 크기의 잔여 오차 영역을 의미합니다. 본 챌린지는 파일 크기 마킹의 오차를 악용하므로 파일 슬랙을 분석해야 합니다.

## 8. 학습 포인트
* **논리 크기 vs 물리 크기**: 컴퓨터 과학 및 스토리지 아키텍처에서 데이터를 블록/클러스터 단위로 주소 배정할 때 필연적으로 생기는 유휴 디스크 조각 영역의 구조를 파악합니다.
* **저수준 디스크 포렌식**: OS 커널 레벨의 파일 API 필터링을 우회하여 디스크 이미지를 직접 이진 바이트 단위로 복제 및 카빙하여 원시 증거를 발굴하는 원리를 터득합니다.
