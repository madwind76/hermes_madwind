# 변조된 BMP 헤더 및 오프셋 수동 복원 (Corrupted BMP File Header & Pixel Data Offset)

> **난이도**: 초중급  
> **소요 시간**: 25~30분  
> **참고 picoCTF 문제**: 손상된 비트맵 이미지 헤더 수리 및 디코드 (BMP File Header 분석)

## 1. 배경 시나리오
공격자가 기밀을 유출하기 위해 스테가노그래피 또는 텍스트 합성을 적용해 제작했던 비트맵 이미지 파일 `corrupted.bmp`가 수집되었습니다. 공격자는 사건 발생 후 보안 스캐너의 이미지 렌더링 및 카빙 인식을 인위적으로 방해하기 위해, BMP 파일의 매직 바이트(`BM`)는 그대로 둔 채 **픽셀 데이터가 실제로 시작하는 위치 정보(Offset to Pixel Data)** 속성을 강제로 훼손(`00 00 00 00` 또는 `FF FF FF FF`)하여 이미지 뷰어 가동을 무력화해 두었습니다. 전형적인 24비트 RGB 비형식 BMP 파일 헤더 명세(총 54바이트 헤더) 구조를 기반으로 오염된 오프셋 바이트 데이터를 수작업 패치(Hex Patching)하여, **복구된 이미지 화면 내에 기재된 최종 플래그**를 획득하십시오.

## 2. 제공 파일
* `corrupted.bmp` (픽셀 시작 바이트 지시자 오프셋 변수가 변조되어 열리지 않는 BMP 파일)

## 3. 문제 목표
윈도우 표준 비트맵(BMP) 포맷 구조(14바이트 File Header와 40바이트 DIB/BITMAPINFOHEADER의 구성 규격 및 오프셋 10~13 위치에 명시되는 4바이트 리틀엔디언 픽셀 오프셋 변수의 연산 방식)를 이해하고, 이를 헥스 패칭하여 이미지를 재구축합니다.

## 4. 의도한 풀이 흐름
1. **바이너리 레이아웃 구조 분석**:
   * 제공된 `corrupted.bmp` 파일을 헥스 에디터로 로드합니다.
   * 선두 오프셋 0~1 위치에서 매직 지시자인 `42 4D` (아스키 문자 `BM`)가 정상 존재함을 확인합니다.
   * 이어서 오프셋 10~13 (최초 파일 헤더의 마지막 4바이트 필드)을 검사합니다. 현재 `00 00 00 00` 또는 유효하지 않은 임의 바이트(`FF FF FF FF`)로 훼손되어 비트맵 이미지 리더가 픽셀 어레이의 시작점을 잡지 못함을 인지합니다.
2. **픽셀 데이터 오프셋 수동 복원**:
   * 일반적인 24비트(True Color) BMP 형식은 추가 컬러 테이블(Color Palette)이 불필요하므로, 14바이트의 Bitmap File Header와 40바이트의 DIB Header (BITMAPINFOHEADER) 뒤에 즉시 실질 픽셀 데이터가 뒤따릅니다.
   * 따라서 헤더의 총 바이트 수인 **54바이트** 위치가 곧 픽셀 데이터의 시작 오프셋이 됩니다.
   * 54를 16진수로 변환하면 `0x36` 입니다.
   * 4바이트 리틀 엔디언 규격으로 변환하여 오프셋 10~13 위치에 바이트 스왑 값인 **`36 00 00 00`** 을 기입합니다.
   * 패치한 바이너리를 `rebuilt_flag.bmp` 이름으로 저장합니다.
3. **이미지 렌더링 및 플래그 획득**:
   * 복구된 `rebuilt_flag.bmp` 파일을 실행하여 렌더링을 확인합니다.
   * 에러가 해결되어 출력되는 비트맵 단색 백그라운드 이미지 속에 적힌 최종 플래그를 식별합니다:
     `picoCTF{bmp_pixel_0ffs3t_hdr_reconstructed}`

## 5. 정답(플래그) 규칙 및 예시
* **플래그 형식**: `picoCTF{<flag_string>}`
* **예시**: `picoCTF{bmp_pixel_0ffs3t_hdr_reconstructed}`

## 6. 출제자 노트 & 파일 제작 가이드
* **아티팩트 제작 방법**:
  1. 기밀 플래그 `picoCTF{bmp_pixel_0ffs3t_hdr_reconstructed}` 텍스트가 명기된 24비트 비압축 BMP 이미지를 빌드합니다. (이때의 기본 구조는 파일 헤더 14바이트 + DIB 헤더 40바이트 = 총 54바이트 헤더 구조를 갖춥니다)
  2. 헥스 에디터로 이미지를 로드한 뒤, 오프셋 10~13 위치에 기록되어 있던 `36 00 00 00` 바이트를 전부 `00 00 00 00`으로 덮어써서 손상시킵니다.
  3. 변조된 이미지를 `corrupted.bmp` 파일로 최종 저장하여 수검자용 배포 아티팩트로 제공합니다.
* **출제 포인트**: 
  * 미디어 파일 포맷(BMP File Specifications)의 헤더 세부 크기 매커니즘을 파악하고, 손상된 메타데이터를 구조적 바이트 연산에 맞추어 수작업 패칭(Little Endian Integer Patching)함으로써 렌더링 엔진의 작동 조건을 정상으로 환원해내는 기본 바이너리 포렌식 기법을 연마하게 합니다.

## 7. 트러블슈팅 및 힌트
* **Q. DIB 헤더 크기 변수와 픽셀 데이터 시작 오프셋의 상관관계는 무엇인가요?**
  * A. BMP 포맷의 DIB 헤더 크기는 오프셋 14~17 위치의 4바이트 변수(통상 BITMAPINFOHEADER 기준 40, 즉 `28 00 00 00`)에 기록됩니다. 따라서 픽셀 오프셋은 기본적으로 `14(File Header 크기) + DIB Header 크기(본 문제에서는 40) = 54(0x36)` 공식을 따릅니다. 만약 DIB 헤더의 크기가 다른 형태(예: BITMAPV5HEADER 등 124바이트)로 구현되어 있다면, 픽셀 오프셋 바이트 값 역시 그에 맞춰 연동 변경되어 기입되어야 정상이므로 항상 DIB 크기 필드를 병행 관찰하는 것이 정밀 진단에 유효합니다.
* **Q. CyberChef를 사용하여 이 수리 작업을 수행할 수도 있나요?**
  * A. 네, 그렇습니다. CyberChef의 `Find / Replace` 모듈을 사용하거나, `Hexdump` 화면에서 특정 영역을 수정하여 다운로드할 수 있어 유용하게 응용 가능합니다.

## 8. 학습 포인트
* **BMP 파일 이진 레이아웃 명세**: 파일 헤더 및 DIB 헤더의 각 필드 오프셋과 24비트 비트맵 픽셀 위치 정의 스키마를 완벽히 이해합니다.
* **이진 헤더 패칭(Hex Patching)**: 이미지 판독 오류를 교정하기 위해 바이트 순서(Endianness)를 조절하고 구조에 부합하는 바이트 값을 주입해 정상 환원하는 능력을 배양합니다.
