# 파이썬 컴파일 바이너리 내 컴파일 타임스탬프 시간 추출 (Python pyc Compilation Timestamp)

> **난이도**: 초급  
> **소요 시간**: 15~20분  
> **참고 picoCTF 문제**: 파이썬 컴파일 바이너리 컴파일 타임스탬프 규명 (Python .pyc Header Timestamp 분석)

## 1. 배경 시나리오
공격자가 사내 서버에 몰래 기동 및 상주시켰던 백도어 스크립트인 `agent.pyc` 파일이 발견되었습니다. 침해 분석팀은 이 스크립트의 빌드 시점을 파악해 개발자의 최초 침투 시간대를 정밀하게 특정하려 합니다. 파일시스템 상의 생성/수정 시간(MACB)은 피의자가 타임스톰핑 도구로 변조해 두어 신뢰할 수 없는 상태입니다. 하지만 파이썬 컴파일러(`.pyc` 생성기)는 바이너리 빌드 시, 원본 파일의 최종 수정 시간 정보인 **32비트 유닉스 에포크 타임스탬프(Unix Epoch Time)**를 파일 헤더 특정 영역에 영구 직렬화하여 기록합니다. 획득한 `agent.pyc` 파일의 헤더를 검토하여, **스크립트가 실제 컴파일 수정 기표되었던 원본 시각(플래그)**을 특정하십시오.

## 2. 제공 파일
* `agent.pyc` (파이썬 3 환경에서 정상 컴파일 수립되어 시스템에 상주 중이던 바이너리 파일)

## 3. 문제 목표
파이썬 컴파일 `.pyc` 파일 헤더의 16바이트 구조(Magic Number 4바이트, Bitfield 4바이트, Modification Time 4바이트, File Size 4바이트)를 파악하고, 오프셋 8~11 위치의 리틀엔디언 4바이트 데이터를 10진수 Unix Epoch 정수로 변환한 뒤 인간 가독 날짜 포맷으로 변환합니다.

## 4. 의도한 풀이 흐름
1. **헤더 오프셋 분석**:
   * 제공된 `agent.pyc` 파일을 헥스 에디터로 엽니다.
   * 파이썬 3 규격상 헤더의 오프셋 구조를 나열해 관찰합니다:
     * `0x00 ~ 0x03`: 매직 넘버 (4바이트)
     * `0x04 ~ 0x07`: 빌드 옵션 플래그 비트 필드 (4바이트)
     * `0x08` ~ `0x0B`: 수정 시각 유닉스 에포크 타임 (4바이트 리틀엔디언)
     * `0x0C` ~ `0x0F`: 원본 파일 크기 (4바이트 리틀엔디언)
2. **에포크 타임 바이트 카빙**:
   * 오프셋 `0x08` 위치부터 `0x0B` 위치까지 기록된 4바이트의 헥사 데이터 값을 획득합니다:
     `3B 4E 40 6A`
3. **리틀엔디언 정수 변환 및 데이트 파싱**:
   * 획득한 헥사 바이트를 리틀엔디언(바이트 스왑) 순서로 나열하여 32비트 정수 값을 계산합니다:
     `3B 4E 40 6A` -> `0x6A404E3B`
   * 16진수 `6A404E3B`를 10진수로 변환합니다:
     `1782643515`
   * 구한 유닉스 에포크 타임스탬프 `1782643515` 정수를 날짜 포맷으로 변환(Python `datetime` 모듈 또는 CyberChef `From UNIX Timestamp` 적용)합니다:
     * 파이썬 예시:
       ```python
       from datetime import datetime, timezone
       print(datetime.fromtimestamp(1782643515, tz=timezone.utc))
       ```
     * 출력 결과: `2026-06-28 10:45:15` UTC
4. **플래그 도출**:
   * 도출된 날짜 시각 정보를 포맷(YYYY-MM-DD_HH:MM:SS)에 맞게 대입해 플래그를 정립합니다:
     `picoCTF{2026-06-28_10:45:15}`

## 5. 정답(플래그) 규칙 및 예시
* **플래그 형식**: `picoCTF{YYYY-MM-DD_HH:MM:SS}`
* **예시**: `picoCTF{2026-06-28_10:45:15}` (모든 시간 연산은 분석가 혼선을 막기 위해 표준 UTC 타임을 기점으로 도출하도록 유도합니다)

## 6. 출제자 노트 & 파일 제작 가이드
* **아티팩트 제작 방법**:
  1. 원본 파이썬 소스 스크립트 파일의 수정 타임을 정확히 유닉스 시간 `1782643515` 초(UTC 기준 `2026-06-28 10:45:15`)에 부합하도록 운영체제 시스템 시각을 조정하거나 스크립트로 파일 타임을 덮어씁니다:
     `touch -d "2026-06-28 10:45:15 UTC" agent.py`
  2. 파이썬 컴파일 명령을 구동해 바이트코드를 빌드합니다:
     `python3 -m py_compile agent.py`
     (이 경우 생성기 엔진이 `agent.py` 파일의 실제 수정 시간인 1782643515 정수 값을 리틀엔디언 바이트인 `3B 4E 40 6A` 형태로 헤더 오프셋 8 위치에 자동 직렬화 기입합니다)
  3. 캐시 디렉터리에서 생성된 `agent.cpython-XXX.pyc` 파일을 확보하고 이름을 `agent.pyc`로 정리하여 배포용 아티팩트로 지정합니다.
* **출제 포인트**: 
  * 파일시스템 MACB 시간 속성이 안티 포렌식 기법에 의해 위장 및 덮어씌워진 극단적인 침해 현장에서, 실행 바이너리(.pyc)의 내부 메타데이터 헤더에 기록되는 원본 유닉스 에포크 타임(Unix Epoch Time Forensics)을 헥사 레벨에서 추출하여 실제 침투 행위 연대기를 정확하게 규명하는 기술적 조사 프로세스를 교육합니다.

## 7. 트러블슈팅 및 힌트
* **Q. 파이썬 2 환경에서 빌드된 .pyc 파일의 헤더 구조도 동일한가요?**
  * A. 아닙니다. 파이썬 2.x 대역에서는 헤더 크기가 8바이트로 구성되어 매직 넘버 4바이트 및 유닉스 에포크 타임 4바이트만 존재하고 비트 필드나 파일 크기 슬라이스가 부재합니다. 본 시나리오에 적용된 파이썬 3 환경에서는 파일 크기와 컴파일 옵션 비트 필드가 덧붙여져 총 16바이트 크기의 헤더를 갖추고 있으므로 분석 오프셋 매핑 시 혼선이 없어야 합니다.
* **Q. 유닉스 에포크 타임은 시스템 로컬 타임존(Timezone)에 영향을 받나요?**
  * A. 유닉스 에포크 타임스탬프(Unix Epoch Time)는 영국 그리니치 표준시인 `1970-01-01 00:00:00 UTC`를 기점으로 흐른 절대적인 누적 초(Seconds) 단위를 정수로 표현하므로, 시스템의 로컬 시간 설정(KST, EST 등)과 관계없이 항상 고정 불변의 절대값을 지닙니다. 다만 이를 날짜 텍스트로 변환하여 읽을 때는 로컬 시스템의 타임존에 따라 시간 단위가 시프트되어 출력될 수 있으므로, 국제 침해 사고 분석 시에는 항상 표준시(UTC)로 명기 대조하여 공유하는 것이 원칙입니다.

## 8. 학습 포인트
* **Python 3 바이트코드 헤더 명세**: 컴파일 바이너리 파일 내부의 저수준 메모리 레이아웃 및 16바이트 구조체 정의 사양을 이해합니다.
* **에포크 타임(Epoch Time) 역연산**: 리틀엔디언 헥사 데이터를 32비트 정수로 계산하고, 이를 표준시(UTC) 기준 날짜 문자열로 복구해 포렌식 시간 증적을 확보하는 절차를 습득합니다.
